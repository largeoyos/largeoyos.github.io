### 第一幕:URL 与 DNS 解析

你在浏览器输入 `https://www.baidu.com`,这是一个 **URL**,结构:

```
https://www.baidu.com:443/index.html
  协议     域名      端口    路径
```

但计算机不认识 `www.baidu.com`,它只认 IP 地址。于是浏览器第一件事:**查询 DNS,把域名翻译成 IP**。

#### 1.1 DNS(Domain Name System,域名系统)

**作用**:域名 ↔ IP 地址的转换。

**端口号**:53(UDP 为主,TCP 为辅)

- UDP:普通查询,快
- TCP:响应数据 > 512 字节(如区域传输)时使用

#### 1.2 域名的层级结构(**必考**)

域名是**树状分层**的,从右往左读:

```
www . baidu . com .
 ↑      ↑      ↑  ↑
主机  二级   顶级 根
域名  域名   域名 域
```

- **根域**:最右端的 `.`(平时省略)
- **顶级域名(TLD)**:`com`、`org`、`net`、`cn`、`jp`
- **二级域名**:`baidu`、`google`
- **主机名**:`www`、`mail`

#### 1.3 DNS 服务器的分层

|级别|职责|
|---|---|
|**根 DNS 服务器**|全球共 13 组,知道顶级域服务器的地址|
|**顶级域 DNS 服务器**|管理 `.com`、`.cn` 等|
|**权威 DNS 服务器**|管理具体域名(如百度自己的 DNS)|
|**本地 DNS 服务器**|ISP 提供,离用户最近,带缓存|

#### 1.4 两种查询方式(**高频考点**)

**① 递归查询**:本地 DNS 必须给客户端一个最终答案。客户端只问一次。

**② 迭代查询**:每个 DNS 服务器如果不知道答案,就告诉查询者"你去问下一个",查询者自己去问。

📌 **典型场景**:

- **客户端 → 本地 DNS**:递归(我就要答案)
- **本地 DNS → 根、顶级、权威**:迭代(你们一步步指路)

#### 1.5 完整 DNS 查询过程(**必考画图**)

```
你的电脑 ──①查 www.baidu.com──→ 本地DNS(递归)
                                      │
                                      │②问根
                                      ├────→ 根DNS
                                      │←──③ 去问 .com 服务器
                                      │
                                      │④问 .com
                                      ├────→ 顶级DNS
                                      │←──⑤ 去问百度的权威DNS
                                      │
                                      │⑥问百度权威
                                      ├────→ 权威DNS
                                      │←──⑦ 110.242.68.66
                                      │
你的电脑 ←──⑧ 110.242.68.66 ───── 本地DNS
```

#### 1.6 DNS 缓存

每一层都有缓存,TTL 到期才重新查询。这就是为什么你访问过的网站下次打开快很多。

📌 **DNS 劫持**:黑客篡改本地 DNS 的解析结果,把你导向假网站。HTTPS 可以在一定程度上防御(后面讲)。

### 第二幕:TCP 三次握手建立连接

好,浏览器现在拿到了百度的 IP `110.242.68.66`。接下来要做什么?

**建立 TCP 连接**(因为 HTTPS 基于 TCP)。

回顾第五章:你的电脑和百度服务器进行**三次握手**:

```
你 ──SYN, seq=x──────────→ 百度
你 ←SYN+ACK, seq=y, ack=x+1── 百度
你 ──ACK, seq=x+1, ack=y+1──→ 百度
```

这个过程中,中间要经历:

- **ARP**(第四章):找到默认网关的 MAC 地址
- **IP 路由**(第四章):跨越数十台路由器从东京到百度机房
- **以太网/WiFi**(第三章):每一跳的物理传输

三次握手完成后,你的电脑和百度服务器之间有了一条**可靠的字节流通道**。

---

### 第三幕:TLS 握手(HTTPS 特有)

因为你访问的是 **HTTPS**(不是 HTTP),TCP 连接建立后,还要做 **TLS 握手**来协商加密。

#### 3.1 HTTP vs HTTPS(**必考辨析**)

||HTTP|HTTPS|
|---|---|---|
|安全性|**明文传输**,可被窃听篡改|**加密传输**|
|端口|80|443|
|协议栈|HTTP / TCP|HTTP / **TLS** / TCP|
|性能|快|略慢(多一次握手)|

#### 3.2 TLS 握手做了什么(**简化版**)

1. **客户端 Hello**:我支持这些加密算法
2. **服务器 Hello**:我们用 XX 算法,这是我的**数字证书**(含公钥)
3. **客户端验证证书**:确认服务器身份(防钓鱼)
4. **协商会话密钥**:用公钥加密一个随机数发给服务器,后续通信用这个对称密钥加密

📌 **核心思想**:**非对称加密**(RSA)传递密钥,**对称加密**(AES)传输数据。前者安全但慢,后者快但需先安全传递密钥。

#### 3.3 数字证书

- 由 **CA(证书颁发机构)** 签发
- 证书里有:域名、公钥、有效期、CA 签名
- 浏览器内置了信任的根 CA,可验证证书真伪

📌 这就是为什么 HTTPS 比 HTTP 安全——不仅加密,还**验证对方身份**。

### 第四幕:HTTP 请求与响应(**重头戏**)

TCP+TLS 都建好了,浏览器终于可以发 HTTP 请求了。

#### 4.1 HTTP 特点(**必考**)

1. **无状态**:服务器不保存客户端任何信息
2. **基于请求/响应**:客户端发,服务器应
3. **基于 TCP**:可靠传输
4. **灵活**:支持多种数据类型(HTML、图片、视频)

📌 "无状态"是什么意思?——服务器处理完一个请求后就"忘了"你是谁,下一次请求还要重新自我介绍。这是优点(服务器简单、可扩展),也是缺点(需要 Cookie/Session 补救)。

#### 4.2 HTTP 报文格式

**请求报文**:

```
GET /index.html HTTP/1.1        ← 请求行
Host: www.baidu.com              ← 首部行
User-Agent: Chrome/120
Accept: text/html
Cookie: BAIDUID=xxx
                                 ← 空行
(请求体,GET 通常为空)            ← 实体主体
```

**响应报文**:

```
HTTP/1.1 200 OK                  ← 状态行
Content-Type: text/html
Content-Length: 2048
Set-Cookie: ...
                                 ← 空行
<html>...</html>                  ← 响应体
```

#### 4.3 HTTP 请求方法(**必背**)

|方法|作用|特点|
|---|---|---|
|**GET**|获取资源|参数在 URL,幂等、无副作用|
|**POST**|提交数据|参数在 body,非幂等|
|**PUT**|上传/替换资源|幂等|
|**DELETE**|删除资源|幂等|
|**HEAD**|只取首部,不要 body|用于检测资源存在|
|**OPTIONS**|询问服务器支持哪些方法|跨域预检|

📌 **GET vs POST 辨析**(经典面试题):

- GET 参数在 URL(长度受限、明文暴露),POST 在 body
- GET 幂等(多次请求结果相同),POST 可能每次不同
- GET 可缓存、可收藏,POST 不行
- **本质上都是 TCP 请求,这些"区别"是约定,不是技术限制**

#### 4.4 HTTP 状态码(**必背**)

|分类|含义|常见|
|---|---|---|
|**1xx**|信息性|100 继续|
|**2xx**|成功|**200 OK**、204 无内容|
|**3xx**|重定向|**301 永久重定向**、**302 临时重定向**、304 未修改(缓存)|
|**4xx**|客户端错误|**400 请求错误**、**401 未授权**、**403 禁止**、**404 未找到**|
|**5xx**|服务器错误|**500 内部错误**、**502 网关错误**、**503 服务不可用**|

📌 记忆口诀:

- 200 成功;301/302 改地方了;404 找不到;500 服务器炸了

#### 4.5 Cookie 与 Session(**经典考点**)

因为 HTTP 无状态,需要额外机制让服务器"记住"用户。

**Cookie**(客户端存储):

1. 第一次访问,服务器响应中带 `Set-Cookie: id=abc123`
2. 浏览器把 Cookie 存起来
3. 以后每次请求,自动带上 `Cookie: id=abc123`
4. 服务器通过 Cookie 认出你

**Session**(服务端存储):

- 服务器为每个用户分配 Session ID
- Session 数据存在服务器,Cookie 只存 ID
- 更安全,但消耗服务器内存

#### 4.6 HTTP 版本演进(**了解**)

|版本|特点|
|---|---|
|HTTP/1.0|每次请求都要新建 TCP 连接(短连接)|
|**HTTP/1.1**|**持久连接**(Connection: keep-alive),管道化|
|HTTP/2|二进制分帧、多路复用、头部压缩|
|HTTP/3|基于 **QUIC**(UDP),彻底抛弃 TCP|

📌 **HTTP/1.1 持久连接**:一次 TCP 连接上可发多次 HTTP 请求,省去每次三次握手的开销。这是**高频考点**。

---

### 第五幕:浏览器渲染与后续请求

服务器返回一个 HTML 页面,浏览器解析时发现里面还有 `<img>`、`<script>`、`<link>`——这些资源都要**再次发起 HTTP 请求**去拿。

现代网页打开一次,往往要发几十甚至上百个 HTTP 请求。HTTP/1.1 的持久连接和 HTTP/2 的多路复用就是为优化这个场景。

### 第六幕:其他应用层协议(快速过)

访问网页的流程讲完了,再补充几个考试常考的其他应用层协议。

#### 6.1 FTP(文件传输协议)

- 端口:**20(数据)、21(控制)**
- 基于 **TCP**
- **两个连接**:控制连接传命令,数据连接传文件
- 两种模式:**主动模式(PORT)** 和 **被动模式(PASV)**

📌 易错:FTP 用两个端口是经典考点。

#### 6.2 TFTP(简单文件传输协议)

- 端口:**69**
- 基于 **UDP**
- 用于小文件传输(如路由器启动配置)

#### 6.3 电子邮件系统(**考试常考**)

**架构**:

```
发送方 ──SMTP──→ 发送邮件服务器 ──SMTP──→ 接收邮件服务器 ──POP3/IMAP──→ 接收方
```

**三大协议**:

|协议|端口|作用|
|---|---|---|
|**SMTP**(简单邮件传输)|**25**|**发送**邮件,服务器间中转|
|**POP3**(邮局协议)|**110**|**接收**邮件,下载到本地后删除服务器副本|
|**IMAP**(交互式邮件访问)|**143**|**接收**邮件,保留在服务器,支持多端同步|

📌 **必考辨析**:

- **SMTP 只能发不能收**
- POP3:下载就删,早期用,适合单终端
- IMAP:保留服务器,适合多终端(手机、电脑都能看)

**邮件格式扩展 MIME**:

- 原 SMTP 只能传 ASCII 文本
- **MIME**(多用途互联网邮件扩展)允许传图片、视频、附件
- 通过 Base64 编码把二进制数据转成 ASCII

#### 6.4 P2P(Peer-to-Peer)

**传统 C/S 架构**:客户端-服务器,服务器压力大 **P2P 架构**:节点之间互相共享,无中心服务器

典型应用:

- **BitTorrent**(BT 下载):文件切片,多节点并行下载
- 早期的 Napster、eMule
- 区块链(比特币、以太坊)本质也是 P2P

📌 **P2P 优势**:下载人越多越快(传统 C/S 恰恰相反)。

#### 6.5 DHCP(回顾)

第四章提过,这里串一下:

- 你的电脑一开机没有 IP
- 发送 DHCP Discover 广播
- DHCP 服务器分配 IP、掩码、网关、DNS 服务器地址
- 端口:服务器 67,客户端 68,基于 UDP

### 🌐 完整流程总结(全书串联)

现在把你"访问 [www.baidu.com"的全过程串起来](http://www.baidu.com%22%E7%9A%84%E5%85%A8%E8%BF%87%E7%A8%8B%E4%B8%B2%E8%B5%B7%E6%9D%A5):

```
【应用层】
  ① DHCP 获取本机 IP、网关、DNS 服务器(UDP 67/68)
  ② DNS 查询域名 → 拿到百度 IP(UDP 53)
  ③ 浏览器构造 HTTP 请求

【传输层】
  ④ TCP 三次握手(与百度 443 端口)
  ⑤ TLS 握手协商加密(HTTPS)
  ⑥ HTTP 请求/响应通过 TCP 字节流传输

【网络层】
  ⑦ 每个 TCP 报文段封装为 IP 数据报
  ⑧ 查路由表,沿 BGP/OSPF 构建的路径跨越众多 AS
  ⑨ ARP 查网关 MAC(每跳都可能用)

【数据链路层】
  ⑩ IP 数据报封装为以太网帧
  ⑪ CSMA/CD(有线)或 CSMA/CA(无线)访问信道

【物理层】
  ⑫ 比特流通过双绞线、光纤、电磁波传输

【完成后】
  ⑬ 响应 HTML 返回,浏览器渲染
  ⑭ 发现还有图片/CSS/JS → 复用 TCP 连接继续请求
  ⑮ 用完后 TCP 四次挥手关闭连接
```

这就是你每一次打开网页,电脑都在飞快做的事。

**题 1(简答)**:简述 DNS 递归查询与迭代查询的区别。

<details> <summary>参考答案</summary>

递归查询:查询方要求被查询方必须给出最终答案,查询方只问一次。 迭代查询:被查询方若不知道答案,返回下一个该问的服务器,查询方继续自己去问。 通常"客户端→本地DNS"用递归,"本地DNS→各级服务器"用迭代。

</details>

<details> <summary>参考答案</summary>

D。DHCP 基于 UDP。

</details>

**题 3(简答)**:HTTP 是无状态的,那网站如何"记住"你已登录?

<details> <summary>参考答案</summary>

通过 Cookie/Session 机制。首次登录后,服务器返回 Set-Cookie,浏览器保存;后续请求自动携带 Cookie,服务器据此识别用户身份。

</details>

**题 4(综合)**:描述从输入 URL 到页面加载的全过程,至少涉及 3 个应用层协议。

<details> <summary>参考答案</summary>

SMTP 发邮件,POP3/IMAP 收邮件。POP3 默认下载后删除服务器副本,换设备后服务器上已无邮件;IMAP 保留服务器副本,支持多端同步。

</details>
