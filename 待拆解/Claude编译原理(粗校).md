## 第一节:编译原理:让机器"理解"人类的语言
欢迎来到编译原理的世界我先给你讲一下**编译原理到底是什么、为什么值得学**。

想象一下,你用 C 语言写了这样一行代码:

c

```c
int sum = a + b * 2;
```

计算机的 CPU 实际上只认识二进制机器码(一堆 0 和 1)。那么,这行人类能读懂的代码,是怎么变成 CPU 能执行的指令的?答案就是**编译器(Compiler)**——它是一个翻译官,把"高级语言"翻译成"机器语言"。

编译原理研究的就是:**这个翻译过程是如何实现的?**

一个编译器通常分为几个阶段,你可以把它想象成一条流水线:

1. **词法分析(Lexical Analysis)**:把代码拆成一个个"单词"(token),比如把 `int sum = a + b * 2;` 识别成 `[int] [sum] [=] [a] [+] [b] [*] [2] [;]`。这就像你读英文句子时先识别每个单词。
2. **语法分析(Syntax Analysis)**:把这些单词组织成一棵"语法树",理解它们的结构关系。比如识别出 `b * 2` 应该先算(因为乘法优先级高),再加上 `a`。
3. **语义分析(Semantic Analysis)**:检查意义是否正确。比如 `a` 和 `b` 有没有声明?类型对不对?
4. **中间代码生成与优化**:生成一种更接近机器但又保持可读性的中间表示,并进行优化(比如把 `2*3` 直接算成 `6`)。
5. **目标代码生成**:最终翻译成汇编或机器码。

### 第一课:形式语言——给"语言"一个数学定义

在讲解析器、自动机之前,我们必须先解决一个根本问题:**什么是"语言"?**

#### 从一个直觉问题开始

看下面两段 C 代码,哪些是"合法的 C 代码"?

c

```c
// (1)
int x = 5;

// (2)  
int x 5 =;

// (3)
for for for = int;
```

你一眼就能看出 (1) 合法、(2)(3) 不合法。但是——**编译器是怎么判断的?** 它凭什么说 (2) 不合法?

答案是:C 语言的设计者写下了一套**规则**,规定了什么样的字符序列属于 C 语言。编译器按照这套规则来"审查"你的代码。

所以问题变成:**我们如何精确地写下这套规则?**

#### 字母表、串、语言

编译原理中,我们用三个基本概念来形式化"语言":

**1. 字母表(Alphabet)**,记作 Σ

字母表是一个**有限的符号集合**。比如:

- Σ₁ = {0, 1} —— 二进制字母表
- Σ₂ = {a, b, c, ..., z} —— 英文小写字母表
- Σ₃ = {if, else, while, +, -, =, id, num, ...} —— 某种编程语言的"词法单元"字母表

注意:字母表里的"符号"不一定是单个字符!对编译器来说,`while` 就是一个符号(token)。

**2. 串(String)**,也叫字符串

串是字母表中符号的**有限序列**。比如 Σ₁ = {0,1} 上的串:

- `0`、`1`、`01`、`1100`、`ε`(空串,长度为 0)

串的长度记作 |s|,比如 |1100| = 4,|ε| = 0。

Σ 上**所有**可能串的集合记作 **Σ***(读作"Σ 星"),这包括空串。

**3. 语言(Language)**

**语言就是 Σ* 的一个子集。**

是的,就这么抽象。一种"语言"无非是:"在所有可能的字符串中,挑出来一些,说它们属于我这个语言"。

举例:

- 语言 L₁ = {所有偶数长度的 01 串} = {ε, 00, 01, 10, 11, 0000, 0001, ...}
- 语言 L₂ = {所有合法的 C 程序} ⊂ (ASCII 字符)*
- 语言 L₃ = {所有能被 3 整除的二进制数的二进制表示} = {0, 11, 110, 1001, 1100, ...}

#### 关键洞见

**"一种编程语言"在数学上,就是"所有合法源程序的集合"。**

C 语言 = 所有合法 C 程序构成的集合(这个集合是无穷大的!)

那么编译器的**第一个任务**就变成了:给定一个字符串 s,判断 s 是不是属于 L(C 语言)。这个问题叫做**识别问题(Recognition Problem)**。

#### 串上的运算

我们定义两种基本运算,后面会反复用到:

**连接(Concatenation)**:两个串首尾相连

- 如果 x = "abc", y = "de",则 xy = "abcde"
- 对任何串 s:sε = εs = s(空串是连接的单位元)

**幂运算**:

- s⁰ = ε
- s¹ = s
- s² = ss
- sⁿ = s·sⁿ⁻¹

#### 语言上的运算

因为语言是集合,所以集合的运算都能用:

|运算|符号|含义|
|---|---|---|
|并|L ∪ M|属于 L 或属于 M 的串|
|连接|LM|{xy \| x ∈ L, y ∈ M}|
|幂|Lⁿ|L 自己连接 n 次|
|克林闭包|L*|L⁰ ∪ L¹ ∪ L² ∪ ... (任意多次,包括零次)|
|正闭包|L⁺|L¹ ∪ L² ∪ ... (至少一次)|

**举个具体例子**,设 L = {a, b},M = {0, 1}:

- L ∪ M = {a, b, 0, 1}
- LM = {a0, a1, b0, b1}
- L² = LL = {aa, ab, ba, bb}
- L* = {ε, a, b, aa, ab, ba, bb, aaa, ...} —— 所有由 a、b 组成的串(含空串)

---

### 为什么这些抽象定义重要?

你可能觉得:"这不就是把直观的东西写复杂了吗?"

其实不是。**一旦"语言"被定义为集合,我们就能用数学方法来研究它了**。比如:

- 两种语言的交集还是语言吗?(是)
- 所有"语言"的集合有多大?(不可数无穷大)
- 能用有限规则描述的语言有哪些?(这就引出了**乔姆斯基文法层级**,下次课讲)
- 哪些语言能被某种机器识别?(这就引出了**自动机理论**)

## 第二节:正则表达式与文法

### 一个核心问题

上节课我们说"语言是 Σ* 的子集"。但马上有个问题:**怎么把一个语言写下来?**

比如 L = {所有偶数长度的 01 串},这是一个无穷集合,你不可能把所有元素列出来。我们需要一种**有限的描述方式来描述无限的语言**。

这就引出了两种最重要的描述工具:**正则表达式** 和 **文法**。

---

### Part 1:正则表达式(Regular Expression)

你应该在编程中用过 regex(比如 Python 的 `re` 模块)。但你可能不知道:**regex 不是发明出来"匹配字符串"的工具,它本质上是一种数学符号系统,用来描述特定类型的语言**。

#### 形式定义

给定字母表 Σ,正则表达式按以下规则**递归定义**:

**基础情况**:

1. `ε` 是正则表达式,表示语言 {ε}(只包含空串)
2. `∅` 是正则表达式,表示语言 ∅(空集)
3. 对每个 a ∈ Σ,`a` 是正则表达式,表示语言 {a}

**归纳情况**:如果 r 和 s 是正则表达式,分别表示语言 L(r) 和 L(s),那么:

4. `r | s` 是正则表达式,表示 L(r) ∪ L(s) —— **选择**
5. `rs` 是正则表达式,表示 L(r)L(s) —— **连接**
6. `r*` 是正则表达式,表示 L(r)* —— **克林闭包(零次或多次)**
7. `(r)` 是正则表达式,表示 L(r) —— **分组**

优先级:`*` > 连接 > `|`(类似 `*` 像乘方,连接像乘法,`|` 像加法)

> 💡 **注意**:这是编译原理教材中的"经典 regex",和 Perl/Python 的 regex 略有不同(后者扩展了很多语法糖,比如 `+`、`?`、`[a-z]`、`\d` 等,但表达能力其实是等价的——只是更方便)。

#### 看几个具体例子

设 Σ = {a, b}:

|正则表达式|描述的语言|
|---|---|
|`a`|{a}|
|`a\|b`|{a, b}|
|`ab`|{ab}|
|`a*`|{ε, a, aa, aaa, ...}|
|`(a\|b)*`|Σ* —— 所有由 a, b 组成的串|
|`a(a\|b)*`|所有以 a 开头的串|
|`(a\|b)*abb`|所有以 abb 结尾的串|

**一个稍复杂的例子**:描述 C 语言的"标识符"(变量名)

字母表是 ASCII。规则是:首字符是字母或下划线,后面字符是字母、数字或下划线。用 regex:

```
letter = [a-zA-Z_]
digit  = [0-9]
id     = letter (letter | digit)*
```

——你看,这就是词法分析器识别变量名的核心规则!

#### 一个稍微烧脑的练习

请你思考:用正则表达式描述"**Σ = {0,1} 上,所有偶数个 0 的串**"。

(提示:把 0 看成"分隔符",中间和两侧可以有任意多个 1)

我直接给答案,你看看能否理解为什么:

```
1* (0 1* 0 1*)*
```

分析:外层 `()*` 表示"成对出现的 0",每对前后都允许夹任意多个 1;最外面的 `1*` 处理"一个 0 都没有"的情况。所以串 `11`、`100110`、`ε`、`0010` 都属于这个语言。

> 💭 **思考**:你能写出"所有 1 的个数是 3 的倍数"的 regex 吗?(可以试试,有点难度,下节课讲自动机的时候你会看到一种系统化的方法)

#### 正则表达式的能力边界 ⚠️

这里要打破一个常见误解:**regex 不是万能的**。

比如这个语言:**L = {aⁿbⁿ | n ≥ 0} = {ε, ab, aabb, aaabbb, ...}**

——"a 和 b 个数相同,a 全在前面,b 全在后面"

**这个语言无法用正则表达式描述!** 直观原因:regex 没有"记忆"或"计数"能力,它无法记住"我已经看了多少个 a"。

这是个重要的理论结果:正则表达式只能描述"**正则语言**(Regular Languages)"这一类——这是表达能力最弱的一类语言。

那么,程序设计语言的语法(比如 C 的嵌套花括号 `{ { { } } }`)——这种结构能用 regex 描述吗?

显然不能,因为它要求"左右括号匹配",本质上和 aⁿbⁿ 一样需要计数。

所以我们需要**更强大的描述工具**——**文法**。

### Part 2:上下文无关文法(Context-Free Grammar, CFG)

文法是一套**生成式规则**,通过反复改写符号来"生成"语言中的串。

#### 形式定义

一个上下文无关文法是一个四元组 **G = (V, T, P, S)**:

- **V**(Variables / Non-terminals,**非终结符**):需要被进一步替换的符号,通常用大写字母表示,如 E、T、F
- **T**(Terminals,**终结符**):最终出现在串中的符号,通常用小写字母或具体符号表示,如 a、b、+、(
- **P**(Productions,**产生式**):形如 `A → α` 的规则,其中 A ∈ V,α 是 (V ∪ T)* 中的串
- **S**(Start symbol,**开始符号**):S ∈ V,生成过程的起点

#### 一个经典例子:算术表达式文法

```
E → E + T | T
T → T * F | F
F → ( E ) | id
```

这里:

- 非终结符 V = {E, T, F}(E=Expression, T=Term, F=Factor)
- 终结符 T = {+, *, (, ), id}
- 开始符号 S = E
- 产生式有 6 条(竖线 `|` 是简写,表示"或")

#### 推导(Derivation):怎么用文法生成串?

从开始符号出发,每一步把一个非终结符**替换**为它的某个产生式右部。直到串中全是终结符为止。

**例子**:用上面的文法推导 `id + id * id`

```
E ⇒ E + T              (用 E → E + T)
  ⇒ T + T              (用 E → T)
  ⇒ F + T              (用 T → F)
  ⇒ id + T             (用 F → id)
  ⇒ id + T * F         (用 T → T * F)
  ⇒ id + F * F         (用 T → F)
  ⇒ id + id * F        (用 F → id)
  ⇒ id + id * id       (用 F → id)
```

成功!这说明 `id + id * id` 是这个文法**生成**的。我们说该文法**定义**的语言就是它能生成的所有串的集合。

#### 语法分析树(Parse Tree)

推导过程可以画成一棵树,叫**语法分析树**:

```
            E
          / | \
         E  +  T
         |   / | \
         T  T  *  F
         |  |     |
         F  F     id
         |  |
         id id
```

**这棵树非常重要!** 它不仅记录了"这个串是合法的",还记录了"它的结构是什么"。比如这棵树清楚地表达了:

- `id + id * id` 应该理解为 `id + (id * id)`
- 因为 `*` 在更深的子树里,**会先被计算**

也就是说,**文法的写法本身就编码了运算优先级**!这是个非常巧妙的设计。

#### 为什么这样写就有了优先级?

仔细看文法:

```
E → E + T | T          (E 由若干个 T 用 + 连接)
T → T * F | F          (T 由若干个 F 用 * 连接)
F → ( E ) | id         (F 是一个 id 或带括号的表达式)
```

层级是:**E(加法层)→ T(乘法层)→ F(原子层)**

加法在最外层,所以最后才被"算"(在树的根附近);乘法在内层,所以先被算。括号通过 F → (E) 重新回到顶层,实现了"括号优先"。

> 这种通过文法层次表达优先级的方法是编译器设计的经典手法。理解了它,你以后看任何编程语言的语法定义都会顺畅很多。

#### 二义性文法(Ambiguous Grammar):一个需要避免的陷阱

如果一个文法**对同一个串能产生两棵不同的语法树**,这个文法叫做**二义性的**。

考虑这个"简化版"算术文法:

```
E → E + E | E * E | ( E ) | id
```

它能生成 `id + id * id`,但是会有**两棵不同的语法树**:

```
树1:                    树2:
    E                       E
  / | \                   / | \
 E  +  E                 E  *  E
 |   / | \             / | \   |
 id E * E             E + E   id
    |   |             |   |
    id  id            id  id

树1理解为 id+(id*id)    树2理解为 (id+id)*id
```

这就麻烦了——同一段代码有两种含义!所以我们说这个文法是**二义性的**。

**编译器设计中,我们要尽量避免使用二义性文法**。前面那个分层写法(E/T/F)就是为了消除二义性,强制运算优先级和结合性。

#### 著名的"悬挂 else 问题"

C 语言中:

c

```c
if (a) if (b) x = 1; else x = 2;
```

这个 `else` 配对哪个 `if`?——文法层面是有歧义的,语言设计者通过"就近匹配"规则来消除歧义。

### Part 3:文法的层次——乔姆斯基谱系

语言学家 **Noam Chomsky** 在 1956 年提出了文法的四个层次,根据产生式的限制条件,语言的表达能力依次增强(其实是减弱限制):

|类型|名称|产生式形式|识别机器|
|---|---|---|---|
|3 型|**正则文法**|A → aB 或 A → a|有限自动机|
|2 型|**上下文无关文法 CFG**|A → α|下推自动机|
|1 型|**上下文相关文法**|αAβ → αγβ|线性有界自动机|
|0 型|**短语结构文法**|α → β|图灵机|

(其中 A 是非终结符,a 是终结符,α、β、γ 是任意串)

**关键观察**:

- **3 型(正则)= 正则表达式能描述的语言**——最弱
- **2 型(CFG)** 能描述大部分编程语言的**语法结构**——主流
- 0 型 = 图灵机能识别的语言 = "可计算"的所有问题——最强

编译器设计中:

- **词法分析**用 3 型文法(正则表达式) → 识别 token
- **语法分析**用 2 型文法(CFG) → 构造语法树
- **语义分析**(类型检查等)需要更强的能力,通常用人工编写的代码,不再用纯文法

> 这是编译原理最优雅的部分之一:**理论上的语言层级,精确对应工程中的处理阶段**。

## 第三节:有限自动机(Finite Automata)

### 一个启发性问题

上节课我们写出了一个正则表达式 `a(a|b)*`,表示"所有以 a 开头的 a、b 串"。

现在问题来了:给你一个字符串 `"abba"`,**你怎么用代码判断它是否属于这个正则表达式描述的语言?**

你可能会想:Python 里 `re.match()` 一行搞定。但底层呢?`re` 模块是怎么实现的?

答案是:**正则表达式会被编译成一种叫"有限自动机"的机器,然后用这个机器逐字符扫描输入**。

这就是我们这节课要讲的内容——**regex 的"执行引擎"**。

---

### Part 1:什么是有限自动机?

#### 直觉理解

想象一个机器人,它只有**有限个状态**,每读一个字符就**改变状态**。当读完所有字符时:

- 如果它处于某个"接受状态"——说"**这个串属于我识别的语言**"
- 否则——说"**不属于**"

就这么简单。这个机器叫**有限自动机(Finite Automaton, FA)**。

#### 一个具体的例子

我们来设计一个自动机,识别"所有以 a 开头的 a、b 串"(也就是 `a(a|b)*`)。

```
        a
    ┌──────→  ((q1))  ←─── a, b
(q0)                    ↓
   │                    │(循环)
   │ b                  │
   ↓                    │
   q2 (死状态)          │
```

状态说明:

- **q0**:初始状态。还没读任何字符。
- **q1**:已经读到第一个字符是 a。这是**接受状态**(用双圈表示)。
- **q2**:第一个字符是 b,不可能再合法了,卡死在这里。

运行过程,输入 `"abba"`:

- 初始在 q0
- 读 'a' → 转到 q1 ✓
- 读 'b' → 留在 q1(因为 q1 有 b 的自环)
- 读 'b' → 留在 q1
- 读 'a' → 留在 q1
- 读完了,当前在 q1(接受状态) → **接受**!

输入 `"bab"`:

- 初始在 q0
- 读 'b' → 转到 q2(死状态)
- 无论后面是什么,永远在 q2
- 读完了,当前在 q2(不是接受状态) → **拒绝**

#### 形式定义

一个**确定性有限自动机(Deterministic Finite Automaton, DFA)**是一个五元组:

**M = (Q, Σ, δ, q₀, F)**

- **Q**:状态的有限集合
- **Σ**:输入字母表
- **δ**:**转移函数**,δ: Q × Σ → Q(给定当前状态和输入字符,确定下一个状态)
- **q₀**:初始状态,q₀ ∈ Q
- **F**:接受状态集合,F ⊆ Q

**关键特点**:"**确定性**"意味着对于每个(状态, 字符)组合,**下一状态唯一确定**。

上面例子形式化:

- Q = {q0, q1, q2}
- Σ = {a, b}
- δ(q0, a) = q1, δ(q0, b) = q2, δ(q1, a) = q1, δ(q1, b) = q1, δ(q2, a) = q2, δ(q2, b) = q2
- q₀ = q0
- F = {q1}

#### DFA 的执行算法(超简单)

python

```python
def simulate_dfa(dfa, input_string):
    state = dfa.q0
    for c in input_string:
        state = dfa.delta[state][c]   # 查表转移
    return state in dfa.F              # 是否在接受状态
```

**时间复杂度:O(n)**,n 是输入长度。每个字符只看一次,只做一次查表。

**这就是为什么词法分析能做到线性时间!** 编译器扫一遍源代码就能把所有 token 识别出来,不需要回溯。

---

### Part 2:不确定性有限自动机(NFA)

#### 为什么要搞个"不确定"的版本?

先看 DFA 的一个"麻烦":构造它有时候很不直观。比如要识别"**以 abb 结尾的串**",你要手动思考每个状态代表什么含义,容易出错。

NFA 放松了两个限制:

1. **同一个(状态, 字符)可以转移到多个状态**(像一个机器人同时变成好几个)
2. **允许 ε 转移**(不读任何字符就能切换状态)

#### NFA 的形式定义

**N = (Q, Σ, δ, q₀, F)**,其中 **δ: Q × (Σ ∪ {ε}) → P(Q)**(P(Q) 是 Q 的幂集,即状态集合的集合)

其他都一样,只有 δ 的返回值变了——返回的是一个**状态集合**,而不是单个状态。

#### 一个例子

识别语言"包含子串 `abb` 的 01ab 串"的 NFA(简化字母表为 {a, b}):

```
       a,b           a         b         b
   ┌─────┐         ┌───┐    ┌───┐    ┌─────┐
   ↓     │         ↓   │    ↓   │    ↓     │
 (q0) ─────────→ (q1) ───→ (q2) ───→ ((q3))
         a
```

注意 q0 上有 `a,b` 的自环,也有一条 `a` 指向 q1 的边——这意味着**读到 'a' 时,q0 可以选择留在 q0,也可以选择去 q1**。这就是"不确定性"。

#### NFA 怎么"运行"?

想象它"同时"走所有可能的路径。只要**存在**一条路径能读完输入并停在接受状态,就说这个串被接受。

输入 `"aabb"`,从 q0 开始:

- 读 'a':同时存在于 {q0, q1}
- 读 'a':从 q0 可去 {q0, q1},从 q1 无法读 a(卡死)→ 合并为 {q0, q1}
- 读 'b':从 q0 只能留在 q0,从 q1 可去 q2 → {q0, q2}
- 读 'b':从 q0 留在 q0,从 q2 可去 q3 → {q0, q3}
- 读完了,q3 ∈ F → **接受**

#### ε 转移的作用

ε 转移允许"瞬间切换"状态,不消耗输入。这让我们可以方便地"粘合"多个小自动机。

比如构造 `a|b` 的 NFA:

```
       ε      a
    ┌────→ q1 ──→ q2
   q0                  ε  
    │                  ┌──→ qf
    │      ε      b    │
    └────→ q3 ──→ q4 ──┘
```

从 q0 出发,**通过 ε 可以选择走上路还是下路**,对应"选择 a 或 b"。

#### 关键定理:NFA 和 DFA 等价

**定理**:对任何 NFA,都存在一个 DFA,接受相同的语言。

也就是说:**NFA 的"表达能力"并没有比 DFA 更强**!它们能识别的语言类是一样的——都是**正则语言**。

那为什么还要 NFA?因为**从 regex 构造 NFA 很容易,构造 DFA 很麻烦**。工程实践中,我们通常:

```
正则表达式 ──Thompson 构造法──> NFA ──子集构造法──> DFA ──最小化──> 最优 DFA
```

---

### Part 3:从正则表达式到 DFA——完整流水线

#### 步骤 1:Thompson 构造法(regex → NFA)

基本思想:**递归地**从 regex 的语法结构,生成对应的 NFA 片段。

**基本模块**:

(1) 对单个字符 `a`:

```
(start) ──a──→ ((end))
```

(2) 对 `ε`:

```
(start) ──ε──→ ((end))
```

**组合规则**:假设 regex r 的 NFA 是 N(r),s 的是 N(s),则:

(3) `r|s`(选择):新建起点和终点,用 ε 连到 r 和 s 的起点,再从 r 和 s 的终点用 ε 连到新终点

```
       ε         ε
  ┌────→ N(r) ────┐
 (new_start)      (new_end)
  └────→ N(s) ────┘
       ε         ε
```

(4) `rs`(连接):把 r 的终点直接用 ε 连到 s 的起点

```
N(r) ──ε──→ N(s)
```

(5) `r*`(闭包):新建起点终点,允许跳过(ε 直达),或循环回 r

```
          ┌──────────ε──────────┐
          │                      ↓
(new_start)─ε─→ N(r) ─ε─→ (new_end)
                 ↑       │
                 └───ε───┘
```

**这个构造是纯机械的**——任何 regex 都能用这几条规则拼出 NFA。状态数最多是 regex 长度的 2 倍(线性)。

#### 步骤 2:子集构造法(NFA → DFA)

核心思想:**让 DFA 的每个状态对应 NFA 的一个状态子集**。

为什么?因为 NFA 运行时"同时存在于多个状态",而这些"多状态组合"的总数是有限的(≤ 2^|Q|)。把每种组合看成 DFA 的一个新状态就行了。

**算法大致步骤**(细节不用死记,理解思路即可):

1. DFA 的初始状态 = NFA 初始状态的 ε-闭包(沿 ε 能到的所有状态)
2. 对每个 DFA 状态 S 和每个输入字符 c,计算:从 S 中任一状态读 c 能到的所有状态,再求 ε-闭包,得到一个新的子集 S'
3. S' 若未出现过,加入 DFA 状态集
4. 重复直到没有新状态产生
5. 包含原 NFA 接受状态的子集 → DFA 的接受状态

**注意**:最坏情况 DFA 的状态数是 NFA 的 2^n,但实际中通常小得多。

#### 步骤 3:DFA 最小化

同一个语言可能对应多个 DFA,有些状态可能是"冗余的"(比如两个状态从任何输入都会走到相同的未来)。

最小化算法(Hopcroft 算法)能把 DFA 压缩到**唯一的最小形式**——状态数最少的等价 DFA。

这对工程很重要:DFA 实现为表格,状态越少,表格越小,内存占用越低。

---

### Part 4:这套理论为什么重要?——回到编译器

现在把拼图拼起来。**词法分析器的完整构造流程**:

```
源代码:  int x = 42;

         ↓ 词法分析器工作:

Token流: [KEYWORD "int"] [ID "x"] [OP "="] [NUM 42] [PUNC ";"]
```

词法分析器怎么来的?

**第一步**:语言设计者用正则表达式定义 token:

```
KEYWORD = if | else | while | int | return | ...
ID      = [a-zA-Z_][a-zA-Z0-9_]*
NUM     = [0-9]+
OP      = \+ | - | \* | / | = | == | ...
```

**第二步**:工具(如 `lex`/`flex`)自动把这些 regex 转成 **NFA**,再转成 **DFA**,再最小化。

**第三步**:DFA 以**转移表**的形式嵌入生成的 C 代码中,运行时逐字符扫描源文件。

这就是为什么 **`flex` 能在几毫秒内处理成千上万行代码**——底层就是一个高效的 DFA 查表。

#### 一个有意思的历史 🕰️

**Ken Thompson**(Unix 和 C 的设计者之一)1968 年发表的论文《Regular Expression Search Algorithm》正是提出了我们讲的 Thompson 构造法。后来他把这个算法实现进了 Unix 的 `grep` 和 `ed` 编辑器——这就是 regex 在编程中普及的起点。

Python 的 `re` 模块用的是**回溯式算法**(更灵活但可能很慢),而 Go 的 `regexp` 模块用的就是 Thompson 的 NFA 算法(严格 O(n))。

---

### 本节要点

|概念|关键性质|用途|
|---|---|---|
|**DFA**|确定性,O(n) 识别|词法分析的"运行机器"|
|**NFA**|允许多路径和 ε 转移|便于从 regex 构造|
|**Thompson 构造**|regex → NFA(线性大小)|工具自动生成|
|**子集构造**|NFA → DFA|让识别高效|
|**最小化**|DFA → 最小 DFA|节省内存|

**核心定理**:正则表达式、NFA、DFA **三者等价**——它们描述的语言类完全相同,都是**正则语言**。

## 第四节:语法分析导论与自顶向下分析

### 先回顾一下,确认我们在同一起跑线

上节课结尾我问了一个问题:为什么 L = {aⁿbⁿ | n ≥ 0} 不能被 DFA 识别?

你不一定要给出完整的证明,但请先在心里想一下:**DFA 有什么"硬限制"**,使得它应付不了这种语言?

想好后继续往下读——我会把这个问题作为这节课的**起点**。

---

### 从一个矛盾出发

假设存在一个能识别 aⁿbⁿ 的 DFA,它有 k 个状态。现在我们输入 `aaa...a`(k+1 个 a)。

DFA 读这 k+1 个 a 的过程中会经过 k+2 个状态(包括初始状态)。但 DFA 只有 k 个状态——根据**鸽笼原理**,**至少有两个位置处于同一个状态**!

假设读完 i 个 a 和读完 j 个 a(i ≠ j)时都在同一个状态 q。那么从 q 继续读 i 个 b 和读 j 个 b,结果必然相同——要么都接受,要么都拒绝。

但是 aⁱbⁱ 应该接受,aʲbⁱ 应该拒绝——**矛盾**!

所以这样的 DFA 不存在。

**核心原因**:DFA 只能记"有限的信息"。它没法记住"到底看了多少个 a",因为要记任意多个数,就需要无穷多状态。

这就引出了我们今天的主角——

### 需要一种"有记忆"的机器

#### 下推自动机(Pushdown Automaton, PDA)

一句话:**PDA = DFA + 一个栈**。

栈就是"无限记忆":想记多少 a 就压入多少符号,后面遇到 b 时一一弹出。读完输入时栈空 → 接受。

有了栈,我们就能识别 aⁿbⁿ、匹配括号、处理嵌套——**这正是编程语言语法的核心特征**。

**理论事实**(不展开证明,知道结论即可):

- **PDA 识别的语言类 = 上下文无关语言(CFL) = CFG 能描述的语言类**

所以 CFG 和 PDA,就像 regex 和 DFA 一样,是**一对搭档**:一个用来"描述",一个用来"识别"。

---

### 但是——工程上我们不直接写 PDA

PDA 在理论上很优雅,但直接写 PDA 很麻烦。编译器工程师发明了更实用的技术,**直接基于 CFG 构造分析器**,分析过程隐含地使用栈。

这些技术分两大类:

|方向|思路|代表方法|
|---|---|---|
|**自顶向下**|从开始符号 S 出发,尝试推导出输入串|递归下降、LL(1)|
|**自底向上**|从输入串出发,反向归约到 S|LR(0)、SLR、LALR、LR(1)|

**这节课讲自顶向下,下节课讲自底向上。**

在开始讲技术细节之前,我想让你先自己思考一下:

> 如果让**你**来设计一个算法,判断"id + id * id"是不是由这个文法生成的:
> 
> ```
> E → E + T | T
> T → T * F | F
> F → ( E ) | id
> ```
> 
> 你会从哪里开始思考?是从 E 开始往下推?还是从 id 这些终结符出发往上归?

这没有标准答案,只是让你体会一下两种方向的差异。带着这个思考我们继续。

### Part 1:自顶向下分析的基本思路

**核心思想**:从开始符号 S 出发,**尝试把它改写成输入串**。

举个极简例子。文法:

```
S → a S b
S → ε
```

输入:`aabb`

推导过程:

```
S                            (当前要匹配 "aabb")
⇒ a S b                      (用 S → aSb。首字符 a 对上了)
⇒ a (a S b) b                (再次用 S → aSb)
⇒ a a S b b                  (现在 S 要匹配空,中间部分对完)
⇒ a a ε b b = aabb           (用 S → ε)
```

✓ 成功!

#### 但这里有个关键问题

每一步选哪条产生式?比如第一步,S 有两条产生式(S → aSb 和 S → ε)——怎么知道选哪个?

**这是自顶向下分析的核心难点**。

我们有两种策略:

**策略 A:瞎猜,猜错就回溯** 试 S → aSb,不行就回来试 S → ε。——这就是**带回溯的递归下降**,正确但可能指数级慢。

**策略 B:看一眼输入,预测选哪个** 如果首字符是 a,必然用 S → aSb;如果是 b 或串已结束,必然用 S → ε。——这就是**预测分析**,线性时间,但要求文法满足一定条件。

工程上我们只用策略 B。那么问题就变成了:**如何根据文法,预先计算出"看到什么字符用哪条产生式"?**

答案涉及两个叫 **FIRST** 和 **FOLLOW** 的集合。我们马上要讲,但在那之前——

第 (1) 问:输入以 `a` 开头,S 必然推成 `AB`,然后 A 用 `A → a` 消掉那个 a。

第 (2) 问:输入以 `b` 开头,S 还是推成 `AB`,但 A 这时要用 `A → ε`(因为 a 不在那)。

### Part 2:FIRST 集和 FOLLOW 集

#### FIRST 集合

**定义**:对任意文法符号(或符号串)α,**FIRST(α) = α 能推导出的所有串中,首字符的集合**。如果 α 能推出 ε,那 ε 也算在 FIRST(α) 里。

**举例**(还是用上面那个文法):

- FIRST(a) = {a} (终结符的 FIRST 就是自己)
- FIRST(ε) = {ε}
- FIRST(A) = {a, ε} (A 能推出 a,也能推出 ε)
- FIRST(B) = {b}
- FIRST(AB) = ? —— 这个稍微复杂,下面讲

**计算规则(非终结符)**:对非终结符 X,看它的每条产生式 X → Y₁ Y₂ ... Yₙ:

- 把 FIRST(Y₁) 中**除 ε 外**的所有符号加入 FIRST(X)
- 如果 ε ∈ FIRST(Y₁),继续看 Y₂,把 FIRST(Y₂) 除 ε 外的加入
- 如果所有 Y₁...Yₙ 都能推出 ε,那么 ε ∈ FIRST(X)

**所以 FIRST(AB)**:

- FIRST(A) = {a, ε},把 {a} 加入 → FIRST(AB) 至少含 {a}
- 因为 A 能推出 ε,继续看 B
- FIRST(B) = {b},加入 → FIRST(AB) = {a, b}
- B 不能推出 ε,停止
- 所以 FIRST(AB) = {a, b}

> 💡 FIRST 集的直觉:"当我要推某个符号时,**第一个字符可能是什么?**"

#### FOLLOW 集合

**定义**:对非终结符 A,**FOLLOW(A) = 在某个句型中紧跟在 A 后面的终结符的集合**。如果 A 可能出现在句子末尾,特殊符号 `$`(输入结束)也加入。

**计算规则**:

- 对开始符号 S:`$` ∈ FOLLOW(S)
- 对每条产生式 B → α A β:把 FIRST(β) 除 ε 外加入 FOLLOW(A)
- 对每条产生式 B → α A,或 B → α A β 且 β 能推出 ε:把 FOLLOW(B) 加入 FOLLOW(A)

> 💡 FOLLOW 集的直觉:"在 A 消失后(推成 ε 或推成别的东西),**紧接下来可能看到什么字符?**"

#### 一个完整的例子

文法:

```
S → A B
A → a | ε
B → b
```

计算 FIRST:

- FIRST(A) = {a, ε}
- FIRST(B) = {b}
- FIRST(S): S → AB,FIRST(AB) = {a, b} → FIRST(S) = {a, b}

计算 FOLLOW:

- FOLLOW(S) = {$}(S 是开始符号)
- 从 S → AB:A 后面是 B,把 FIRST(B)={b} 加入 FOLLOW(A) → FOLLOW(A) = {b}
- 从 S → AB:B 在产生式末尾,把 FOLLOW(S)={} 加入 FOLLOW(B) → FOLLOW(B) = { }

#### 用 FIRST/FOLLOW 做预测

现在我们可以回答那个问题了:**看到什么字符用哪条产生式?**

规则很直接:对非终结符 A 和它的产生式 A → α:

- 如果 FIRST(α) 不含 ε:当前输入字符 ∈ FIRST(α) 时,用这条产生式
- 如果 FIRST(α) 含 ε:当前输入字符 ∈ FOLLOW(A) 时,用这条产生式(意味着 A 该消失)

以 A 为例(A → a 或 A → ε):

- 看到 `a` → 用 A → a(因为 a ∈ FIRST(a) = {a})
- 看到 `b` → 用 A → ε(因为 b ∈ FOLLOW(A) = {b})
- 看到别的 → 错误

### Part 3:LL(1) 文法

#### 什么是 LL(1)?

**LL(1)** 是自顶向下预测分析的"优等生标准":

- 第一个 **L**:从**左**(Left)到右扫描输入
- 第二个 **L**:构造**最左**(Leftmost)推导
- **(1)**:每次看 **1** 个输入字符做预测

一个文法是 LL(1) 的,当且仅当对每个非终结符 A,它的所有产生式 A → α₁ | α₂ | ... | αₙ 满足:

**对任意 i ≠ j:预测集(α_i) ∩ 预测集(α_j) = ∅**

——也就是说,**不会出现两条产生式都想匹配同一个输入字符**的情况。这样一来,"看一个字符就能决策"才成立。

#### 哪些文法不是 LL(1)?——两个常见障碍

**障碍 1:左递归**

产生式形如 `A → A α | β` 的叫**直接左递归**。比如:

```
E → E + T | T
```

从 E 出发,如果选 E → E+T,会无限展开 E → E+T → E+T+T → ... **递归下降永远出不来**。

**解决方法:消除左递归**。上面那条规则可以改写成:

```
E  → T E'
E' → + T E' | ε
```

思路:把 E 先拆出一个 T 头,剩下的"+ T + T + T ..."用 E' 表示成右递归。

**障碍 2:公共左因子**

产生式形如 `A → α β₁ | α β₂`,两条规则开头一样。比如:

```
S → if E then S else S | if E then S
```

看到 `if`,两条都能用——无法根据一个字符决策。

**解决方法:提取左因子**:

```
S  → if E then S S'
S' → else S | ε
```

先读公共部分,再在分叉点决策。

#### 两个思考题(在你继续之前)

**思考 A**:为什么这个文法不是 LL(1)?

```
A → a B | a C
B → b
C → c
```

**思考 B**:刚才我们讲了 `S → aSb | ε` 这个文法。你能心算一下 FIRST(S) 和 FOLLOW(S) 吗?它是 LL(1) 的吗?

先试试,然后再往下看。

---

（答案）

**思考 A**:两条 A 的产生式左部都从 a 开始(有公共左因子),FIRST(aB) = FIRST(aC) = {a},相交不空,所以不是 LL(1)。提取左因子后:`A → a A'`,`A' → B | C`。

**思考 B**:

- FIRST(S) = {a, ε}(因为 S → aSb 贡献 a,S → ε 贡献 ε)
- FOLLOW(S) = {$, b}(S 是开始符号,贡献 $;S → aSb 中 S 后面是 b,贡献 b)
- 检查 LL(1):两条 S 产生式的预测集
    - S → aSb 的预测集 = FIRST(aSb) = {a}
    - S → ε 的预测集 = FOLLOW(S) = {$, b}
    - {a} ∩ {$, b} = ∅ ✓
- **是 LL(1) 的!**

### Part 4:递归下降分析——工程中的实现

理解了 LL(1) 原理,实现起来非常直观:**每个非终结符写一个函数,函数体按产生式右部依次"匹配"**。

用 `S → aSb | ε` 举例,Python 伪代码:

python

```python
input = "aabb$"   # $ 标记结束
pos = 0           # 当前位置

def peek():
    return input[pos]

def consume(c):
    global pos
    if input[pos] == c:
        pos += 1
    else:
        error(f"Expected {c}, got {input[pos]}")

def parse_S():
    if peek() == 'a':           # 预测 S → aSb
        consume('a')
        parse_S()
        consume('b')
    elif peek() in ['$', 'b']:  # 预测 S → ε
        pass                     # 什么都不做,空产生式
    else:
        error("Unexpected token")

# 主程序
parse_S()
if peek() == '$':
    print("Accepted!")
else:
    print("Rejected")
```

**核心观察**:

1. **递归下降分析器的调用栈**,本质上就是 PDA 的那个栈!每一次函数调用相当于把"我还要完成什么任务"压栈。
2. 代码**结构高度反映文法结构**——看代码就能看出文法长什么样。
3. **非常容易手写**,很多真实的编译器前端(包括 GCC 的早期版本、Clang、rustc 等)都用递归下降。

#### 再来一个稍复杂的例子

消除左递归后的算术表达式文法:

```
E  → T E'
E' → + T E' | ε
T  → F T'
T' → * F T' | ε
F  → ( E ) | id
```

递归下降:

python

```python
def parse_E():
    parse_T()
    parse_E_prime()

def parse_E_prime():
    if peek() == '+':
        consume('+')
        parse_T()
        parse_E_prime()
    # else: ε, 什么都不做

def parse_T():
    parse_F()
    parse_T_prime()

def parse_T_prime():
    if peek() == '*':
        consume('*')
        parse_F()
        parse_T_prime()
    # else: ε

def parse_F():
    if peek() == '(':
        consume('(')
        parse_E()
        consume(')')
    elif peek() == 'id':
        consume('id')
    else:
        error("Expected ( or id")
```

输入 `id + id * id $`,跟着跑一遍,你会发现函数调用链精确对应了这个表达式的语法树结构。

---

### 本节要点回顾

|概念|要点|
|---|---|
|**自顶向下**|从 S 出发尝试推导输入,本质上是"预测 + 匹配"|
|**FIRST 集**|一个符号串可能的"首字符"集合|
|**FOLLOW 集**|非终结符后面可能跟什么终结符|
|**LL(1) 条件**|每个非终结符的不同产生式的预测集互不相交|
|**左递归**|必须消除,否则递归下降死循环|
|**公共左因子**|必须提取,否则一字符预测不够|
|**递归下降**|每个非终结符一个函数,直接从文法到代码|

#### 一个重要认识

**LL(1) 文法是一个真子集**——不是所有 CFG 都能改造成 LL(1)。比如某些本质二义性的语言、或者需要两个符号才能决策的语言,LL(1) 做不了。对于这些情况,我们需要**更强的分析方法**——这就是下节课的 **LR 分析**(自底向上)。

LR 能处理的文法严格**大于** LL:几乎所有主流编程语言的文法都是 LR 的(常见为 LALR(1),yacc/bison 用的就是这个)。

## 第五节:自底向上分析(LR 分析)

### 先做一个思维实验

上节课我们讲自顶向下:从 S 出发,每一步"猜"一条产生式,试图把 S 展开成输入串。

**现在请你反着想**:

考虑文法:

```
E → E + T | T
T → T * F | F
F → ( E ) | id
```

输入是:`id + id * id`

我不从 E 出发了。**我从输入串本身出发**,我想一步步"把它收回成 E"。也就是说,我要反向使用产生式:不是"E 变成 E+T",而是"看到 E+T,把它收成 E"。

**问题来了**:第一步,我该怎么动?`id + id * id` 里最左边的 `id`——我能不能**立刻**把它收成 F(因为 F → id)?

先停一下,想一想:

> 🤔 **直觉问题**:如果我现在贪婪地看到什么就收什么,可能会出什么问题?
> 
> 比如,如果某个点我看到栈上有 `E + T`,我该立刻把它收成 `E`,还是再等一等看看后面是什么?

想想你的直觉,然后往下读。

---

### 核心矛盾

你可能已经意识到:**"什么时候该收?"是自底向上分析的核心难题**。

- 如果看到 `T` 就立刻收成 `E`(因为 E → T),那后面如果还有 `* F`,就再也没机会用 `T → T * F` 了——**收早了**。
- 如果迟迟不收,可能永远也走不到 E——**收晚了**。

**这个"什么时候该收"的时机判断,就是 LR 分析的核心**,它需要一整套机制来回答。

我先卖个关子:LR 分析用了一个很聪明的设计——**把所有"可能处于的推导状态"打包成一个有限状态机**,用这个状态机来预测"下一步该移还是该收"。听起来抽象?没关系,我们慢慢建立。

### Part 1:移进-归约分析的基本框架

#### 两个基本动作

自底向上分析器用一个**栈**来工作,不断做两件事之一:

**动作 1:移进(Shift)**——把当前输入字符压入栈 **动作 2:归约(Reduce)**——栈顶若干个符号匹配某个产生式右部,就把它们弹出,压入左部非终结符

最终目标:**栈里只剩 S,输入也读完了** → 接受。

#### 手动跟踪一个例子

文法:

```
(1) E → E + T
(2) E → T  
(3) T → T * F
(4) T → F
(5) F → ( E )
(6) F → id
```

输入:`id + id * id $`(`$` 是结束标记)

我先**手工给你演示**一遍(假设我们有某种上帝视角,知道每一步该做什么),你先感受"移进-归约"的节奏:

|步|栈|剩余输入|动作|
|---|---|---|---|
|1|(空)|id + id * id $|移进 id|
|2|id|+ id * id $|归约 F → id|
|3|F|+ id * id $|归约 T → F|
|4|T|+ id * id $|归约 E → T|
|5|E|+ id * id $|移进 +|
|6|E +|id * id $|移进 id|
|7|E + id|* id $|归约 F → id|
|8|E + F|* id $|归约 T → F|
|9|E + T|* id $|**移进 *** ← 注意这里!|
|10|E + T *|id $|移进 id|
|11|E + T * id|$|归约 F → id|
|12|E + T * F|$|归约 T → T * F|
|13|E + T|$|归约 E → E + T|
|14|E|$|接受 ✓|

#### 🔍 关键观察:第 9 步的"抉择"

第 9 步栈顶是 `E + T`——这时候我们**明明可以**用 E → E + T 把它归约成 E(这会得到第 4 步那样的孤零零一个 E)。但我们**没有**,而是选择了**移进 `*`**。

为什么?因为后面跟着 `*`,意味着后面还有乘法,我们应该先处理乘法(让 T 继续增长),再处理加法。

> 💡 **这就是 LR 分析器必须学会的"判断力"**:在移进和归约都可行时,能根据"向前看的符号"做出正确选择。

**问：这种判断力,编译器是怎么获得的?它不能靠直觉啊。**

答案是:把所有可能处于的"分析状态"系统地枚举出来,做成一个有限状态机 —— 这就是接下来要讲的 **LR 自动机**。

---

### Part 2:LR(0) 项目与 LR 自动机

#### LR(0) 项目(Item)

一个**项目**是一个产生式,加上一个圆点 `·`,表示"**分析到哪儿了**"。

比如 E → E + T 这条产生式产生 4 个项目:

```
E → · E + T     (还没开始)
E → E · + T     (刚把 E 压上栈)
E → E + · T     (刚把 + 压上栈)
E → E + T ·     (整个右部都在栈里了,可以归约)
```

**项目的含义**:点左边的部分已经在栈里,点右边的部分我们希望在剩余输入里看到。

**点在最右边**(形如 `A → α ·`)的项目叫**归约项目**——意味着"栈顶这段可以归约"。

#### 项目集(LR(0) 状态)

一个**项目集**是一组项目,代表"我们可能正处于这些分析情形中的任何一个"。

举例:假设我们刚开始,栈是空的。分析器希望看到一个完整的 E。那么我们"可能"处于哪些情形?

- `E' → · E`(我们加一条辅助产生式 E' → E,把原开始符号包起来)

但等等——E 能推成什么?E → E + T 或 E → T。所以为了"开始匹配 E",我们**同时也在**:

- `E → · E + T`
- `E → · T`

T 呢?T 能推成 T * F 或 F,所以也在:

- `T → · T * F`
- `T → · F`

F 呢?F → ( E ) 或 F → id,所以也在:

- `F → · ( E )`
- `F → · id`

这个**自我扩展**过程叫 **闭包(closure)**:只要项目集中有形如 `A → α · B β` 的项目(点后面是非终结符 B),就把所有 `B → · γ` 加进来,直到没有新项目。

所以初始项目集 I₀ 是:

```
I₀: E' → · E
    E  → · E + T
    E  → · T
    T  → · T * F
    T  → · F
    F  → · ( E )
    F  → · id
```

**这一个项目集,就是 LR 自动机的一个状态!**

#### 项目集之间的转移

从 I₀ 出发,如果读到一个符号 X,点会"越过" X,产生新的项目集。这叫 **GOTO(I, X)**。

**举例**:从 I₀ 读到 `id`,哪些项目里的点后面紧跟 `id`?只有 `F → · id`。点越过 id:

```
I₅:  F → id ·
```

这是个**归约项目**——到了这个状态,就该做归约 F → id。

**再举例**:从 I₀ 读到 E,哪些项目点后面是 E?有:

- `E' → · E` → `E' → E ·`
- `E → · E + T` → `E → E · + T`

新集合(还要做闭包,但这两个项目点后都不是非终结符,闭包不增加):

```
I₁: E' → E ·
    E  → E · + T
```

注意 I₁ 里同时有:

- `E' → E ·`(一个**归约项目**,建议归约——但实际上是接受)
- `E → E · + T`(一个**移进项目**,建议移进 +)

这里有两种动作都可能——好在它们看不同的输入符号(`$` vs `+`),所以**不会真冲突**。LR 分析能力的核心,就是区分这些"看起来冲突,其实不冲突"的情况。

#### LR(0) 自动机

把所有可能的项目集枚举出来,状态之间按 GOTO 连接——这就得到一个**有限状态机**。这是 DFA,因为每个(状态, 符号)的下一状态唯一。

对我们这个算术文法,完整构造会得到 12 个状态。我不全列了(构造机械但繁琐),但你要记住**关键一点**:

> **LR(0) 自动机识别的,不是源语言的串,而是"栈上符号的合法前缀"**。
> 
> 也就是:只要栈上的内容是"某个推导过程中能出现的前缀",LR(0) 自动机就能读着它,走到对应状态。

这句话很抽象,我用例子解释:

栈上是 `E + T`——这是"处于 E → E + T · 的分析过程"的一个合法前缀。LR 自动机读 E 到某状态,读 + 到另一状态,读 T 到一个包含 `E → E + T ·` 的状态。

**这个状态本身就告诉分析器:现在栈顶是 E + T,可以归约**。

### Part 3:LR 分析表

把 LR 自动机转成**表格**,就是编译器能用的 **LR 分析表**,分两部分:

**ACTION 表**:ACTION = 移进 / 归约 / 接受 / 错误 **GOTO 表**:GOTO = 下一状态(归约完成后跳转)

#### LR 分析算法

```
栈初始为 [0](初始状态)
循环:
    s = 栈顶状态
    a = 当前输入符号
    查 ACTION[s, a]:
        - 若是"移进 t":把 a 和 t 压栈,输入指针后移
        - 若是"归约 A → β":弹出 2·|β| 个符号(β 的长度两倍,因为状态和符号交替),
                            设此时栈顶状态是 s',把 A 和 GOTO[s', A] 压栈
        - 若是"接受":成功
        - 若是"错误":报错
```

**时间复杂度**:O(n)——每次要么消耗一个输入字符,要么做一次归约,总步数线性。

#### 冲突

当一张 LR(0) 表里,某个 ACTION[s, a] 格子**同时要求移进和归约**,就叫**移进-归约冲突**。如果同一格要求两种不同的归约,叫**归约-归约冲突**。

冲突意味着:**LR(0) 不足以处理这个文法,需要更强的版本**。

### Part 4:LR 家族——从 LR(0) 到 LALR(1)

LR 不是一个方法,而是一**族**方法,表达能力递增:

|方法|决策依据|表达能力|表大小|
|---|---|---|---|
|**LR(0)**|只看状态,不看输入|最弱|小|
|**SLR(1)**|LR(0) + FOLLOW 集判断|中|小|
|**LALR(1)**|折中方案|强|小(= LR(0))|
|**LR(1)**|每个项目带向前看符号|最强|**大**(可能爆炸)|

**工程现实**:

- **LR(1)** 理论上最强,但状态数可能是 LALR 的几十倍,表太大
- **LALR(1)** 把 LR(1) 中"核心相同"的状态合并——**几乎和 LR(1) 一样强,但表和 LR(0) 一样小**
- **LALR(1) 是工业标准**:yacc、bison、早期 GCC 都用它

#### SLR(1) 的核心思想(快速理解)

对归约项目 `A → α ·`,不是无条件归约,而是**只有当前输入符号 ∈ FOLLOW(A) 时才归约**。

这能解决很多 LR(0) 处理不了的冲突,因为"A 后面不可能跟 a"意味着"栈顶虽然匹配了 A 的右部,但现在不是该归约的时候"。

#### LR(1) 项目:带向前看符号

LR(1) 项目形如 `[A → α · β, a]`,多出来的 a 叫**向前看符号**——只有看到 a 时才考虑用这个项目归约。

这比 SLR 更精细:SLR 是对整个文法算 FOLLOW,LR(1) 是对每次具体的推导情境算"下一步可能是什么"。

### Part 5:LL vs LR —— 两种哲学的对比

到这里,你已经掌握了编译器的两大分析技术。我想让你停下来,**自己先想一想**:

> 🤔 **思考问题**:
> 
> 1. LL 和 LR 都是"从左到右读输入、线性时间",那它们最本质的区别是什么?
> 2. 为什么工业界主流(yacc/bison/ANTLR 3/Java 早期编译器)倾向 LR?为什么有些现代编译器(Clang/rustc/GCC 现在)又回到手写递归下降(LL)?

先想一会儿,再往下读。

---

（答案供参考)

**区别 1**:决策时机不同

- LL:**看到产生式的第一个符号之前就得决策**(看当前 token 决定用哪条产生式)——"先承诺,再匹配"
- LR:**看完产生式的整个右部才决策**(栈里攒够了再收)——"先观望,最后归约"

这就是为什么 LR 更强:它能使用"更多的上下文信息"来决策,所以能处理左递归、更宽的文法类等。

**区别 2**:方向的非对称性

- LL:从根向叶(预测)
- LR:从叶向根(归约)——反向推导

**为什么工业界选 LR**:表达能力强,对语言设计者友好(可以自然写左递归的算术表达式),工具自动生成。

**为什么又回归 LL(手写递归下降)**:

1. **错误恢复和错误提示**更灵活(LR 错误信息出了名地差)
2. **调试容易**(就是普通函数调用链)
3. **可以灵活处理上下文依赖**(比如 C 的 typedef 问题,需要词法分析看符号表,纯 LR 很难弄)
4. **性能**:虽然理论上等价,工程上手写代码常比生成的表驱动代码更快

所以现代趋势:**前端手写递归下降,语义和后端用专业工具**。这是工程理性的选择。

|---|---|
|**自底向上**|从输入反向归约到 S,本质是"逆向推导"|
|**移进/归约**|两大基本动作,难点是时机判断|
|**LR(0) 项目**|带圆点的产生式,记录"分析进度"|

|**SLR/LALR/LR(1)**|能力递增,LALR(1) 是工程甜点|
|**LR 比 LL 强**|但手写递归下降更灵活,现代编译器常反而选 LL|

### Part 1:核心思想——文法不仅能"判定",还能"计算"

到目前为止,我们把文法当作一个**判定器**:给一个串,说"合法"或"不合法"。

现在我们要做一个**观念升级**:

> **文法不光能判定语言,还能在判定过程中同时计算一些东西**。

什么叫"计算一些东西"?举一个极简的例子。

#### 一个迷你例子:算一个算式的值

文法:

```
E → E + T | T
T → T * F | F
F → ( E ) | num
```

给定输入 `2 + 3 * 4`,我们不光希望知道它语法合法,**我们还想知道它的值是 14**。

怎么做?给每条产生式加一个"计算规则":

```
产生式                   语义规则
─────────────────────────────────────
E → E₁ + T               E.val = E₁.val + T.val
E → T                    E.val = T.val
T → T₁ * F               T.val = T₁.val * F.val
T → F                    T.val = F.val
F → ( E )                F.val = E.val
F → num                  F.val = num.lexval
```

(这里给同一条产生式里重复出现的非终结符加下标,比如 E₁,是为了区分,因为左边的 E 和右边的 E 是两个不同的"节点"。)

**每个符号带一个属性 `.val`(它的"值")**,每条产生式说明"如何从右部的 val 算出左部的 val"。

#### 跟着走一遍

输入 `2 + 3 * 4`,语法分析建出这棵树:

```
           E
         / | \
        E  +  T
        |   / | \
        T  T  *  F
        |  |     |
        F  F    num(4)
        |  |
       num num
       (2) (3)
```

**从叶子往根算属性**:

- 叶子 num(2).lexval = 2 → 对应 F.val = 2 → T.val = 2 → E.val = 2
- num(3).lexval = 3 → F.val = 3 → T.val = 3
- num(4).lexval = 4 → F.val = 4
- 右子树的 T:T.val = T.val × F.val = 3 × 4 = **12**
- 根 E:E.val = E.val + T.val = 2 + 12 = **14** ✓

这就是一个完整的计算器!它的控制流完全由语法分析驱动,**每归约一次就计算一次**。

#### 这告诉了我们什么?

**一棵语法树可以带着信息**——每个节点挂上属性,根据产生式的语义规则,从子节点的属性计算父节点的属性(或反过来)。

这个思想的威力在于:**只要你能把"某种处理"表达成属性上的规则,那它就能自动嵌入语法分析过程**。

- 求表达式的值 → 属性是 `val`
- **类型检查** → 属性是 `type`
- **生成中间代码** → 属性是 `code`(一段代码串)
- **计算代码的目标地址** → 属性是 `addr`

你会在后面看到,中间代码生成也是用这套机制做的。

---

### Part 2:综合属性 vs 继承属性

属性分两类,理解这个分类对实现很关键。

#### 综合属性(Synthesized Attribute)

**值从子节点传给父节点**——"自下而上"流动。

上面算表达式的值,**val 就是综合属性**:父节点的 val 由子节点的 val 计算。

#### 继承属性(Inherited Attribute)

**值从父节点(或兄弟节点)传给子节点**——"自上而下"或"横向"流动。

举个例子。考虑 C 的变量声明:

```
int x, y, z;
```

文法可以是:

```
D → T L
T → int | float
L → L , id | id
```

现在语义要求:**把类型 T 赋给 L 展开出的每一个 id**。

- L 看不到 T(T 是它的兄弟),但 L 需要知道类型
- 所以我们让 L 有一个**继承属性 `L.inh_type`**,由父节点 D 在产生式 D → T L 里把 T.type 传下来

语义规则:

```
产生式               语义规则
──────────────────────────────────────
D → T L              L.inh = T.type              ← 继承
T → int              T.type = integer
T → float            T.type = float
L → L₁ , id          L₁.inh = L.inh             ← 继承
                     addtype(id.entry, L.inh)
L → id               addtype(id.entry, L.inh)
```

这样建树的同时,**类型信息就从 T 节点"流向"所有的 id 节点**,把它们登记到符号表里。

#### 综合 vs 继承:一张图记住

```
         父 (综合 ← 子)
        / \
      子   子 (继承 ← 父)
```

- **综合属性**:孩子算完传给父亲。**自底向上**分析(LR)天然适合。
- **继承属性**:父亲或兄长传给自己。**自顶向下**分析(LL、递归下降)天然适合。

🤔 你能猜猜为什么吗?想一下:LR 分析是归约时(栈顶有完整右部时)触发语义动作——那时子节点都处理完了,自然方便综合。LL 分析是展开时(展开产生式右部之前)就能定义父的属性,自然方便继承。

这就是**分析策略和语义计算方式的深刻耦合**。

#### S-属性定义和 L-属性定义

工程上我们关心两种"好用"的文法:

- **S-属性定义**:**只有综合属性**。非常适合 LR 分析器(yacc/bison 默认支持)。
- **L-属性定义**:**综合 + 部分受限的继承属性**(继承属性只能依赖左边兄弟或父节点)。适合自顶向下一遍处理。

大多数实际的编译器文法都设计成 L-属性定义,因为它覆盖面广,又能一遍扫完。

---

### Part 3:符号表——语义分析的"记忆"

文法里的**属性**只解决了"信息流动"的问题。但有些信息的作用域跨越了整棵树,比如"一个变量的声明和它的使用可能在树的不同分支"。

比如:

c

```c
int x = 5;
{
    int x = 10;   // 新作用域里的 x
    y = x;        // 这里的 x 是哪个 x?
}
y = x;            // 这里的 x 又是哪个?
```

这种"跨越作用域"的引用关系,用属性传递已经很笨拙了。工程上的做法是:**用一个全局的数据结构——符号表(Symbol Table)**。

#### 符号表是什么

一张(或一叠)表,**每个条目记录一个名字的全部信息**:

|名字|类型|作用域层级|地址/偏移|其他(参数表等)|
|---|---|---|---|---|
|x|int|1|-4|-|
|x|int|2|-8|-|
|printf|function|0|(外部)|(args: char*, ...)|

**常见实现**:

- 哈希表(查找快)
- **嵌套作用域**用**链式的符号表栈**:进入一个新作用域(比如 `{`)就压一个新表,离开(`}`)就弹出。查找时从栈顶向底逐层找,找到最近的那个就返回——这就自然实现了"内层遮蔽外层"。

#### 语义分析时,符号表怎么用

在语法制导翻译中,符号表操作被嵌入到语义规则里:

- 遇到声明(如 `int x;`):`enter(x, int, ...)` ← 插入符号表
- 遇到引用(如 `x + 1`):`lookup(x)` ← 查符号表,取出类型
- 进入块 `{`:新建一层作用域
- 离开块 `}`:弹出一层作用域

**符号表其实是"上下文信息"的载体**,补足了纯 CFG 处理不了的那些"上下文相关"需求。还记得乔姆斯基谱系里的 **1 型文法(上下文相关)**吗?我们并没有真的用 1 型文法去分析——而是**用 2 型文法 + 符号表**来"模拟"上下文相关的检查。这是工程上的漂亮妥协。

---

### Part 4:类型检查——语义分析的主菜

有了属性和符号表,类型检查就水到渠成了。

#### 基本思路

**每个表达式节点都有一个 type 属性**,类型检查的语义规则类似:

```
产生式              类型规则
──────────────────────────────────────────────
E → E₁ + E₂        if (E₁.type == int && E₂.type == int)
                       E.type = int
                   else if (numeric types match via promotion)
                       E.type = (promoted type)
                   else
                       type_error()

E → E₁ [ E₂ ]      if (E₁.type == array(t) && E₂.type == int)
                       E.type = t
                   else
                       type_error()

E → id             E.type = lookup(id.name).type
                   // 如果没找到,报"未声明"错误
```

这样建立语法树的同时,**每个节点的 type 就顺便算出来了**,发现矛盾立刻报错。

#### 类型系统:一个略深的话题

类型检查看起来简单,其实是个大坑。真实语言涉及:

- **类型兼容性**:`int` 能自动转 `float`,但 `int` 能转 `int*` 吗?
- **类型推导**:`auto x = foo();` 里 x 的类型怎么算?
- **多态**:C++ 的模板、Java 的泛型、Rust 的 trait
- **子类型关系**:`Cat` 是 `Animal` 的子类,能不能用 `Cat` 当 `Animal` 用?
- **类型等价**:`struct A { int x; }` 和 `struct B { int x; }` 是"同一个类型"吗?(按名等价 vs 按结构等价)

这些属于**类型理论(Type Theory)**,研究生课会深入。编译原理课通常只讲基础。

### Part 5:一个具体的"微型编译器前端"

把所有东西串起来看一遍。假设我们在处理:

c

```c
int a, b;
a = b + 3;
```

**完整流程**:

**1. 词法分析**:

```
[int] [id "a"] [,] [id "b"] [;]  [id "a"] [=] [id "b"] [+] [num 3] [;]
```

**2. 语法分析**:构造语法树

**3. 语义分析**(在语法分析过程中交织进行):

|动作时机|做了什么|
|---|---|
|规约 D → T L(`int a, b` 归约时)|T.type = int,顺着传下去,把 a 和 b 都加到符号表,类型 = int|
|规约到 `a = b + 3` 中的 E → id(b)|查符号表,取出 b.type = int,E.type = int|
|规约到 `b + 3` 中的 E → num|E.type = int|
|规约到 `b + 3` 中的 E → E + E|两边都是 int,通过,E.type = int|
|赋值 a = ...|查 a.type = int,两边类型一致,通过 ✓|

**4. 生成中间代码**(下一节的内容,这里先剧透):

每条规则除了做类型检查,还可以附带产生一段中间代码串,比如:

```
t1 = b + 3
a = t1
```

这段代码叫**三地址码**,下节课细讲。

### Part 6:退一步看全景

到这里,我们可以把编译器前端的完整架构画出来了:

```
源代码
  │
  ▼
┌──────────────┐
│  词法分析器   │  → token 流
└──────────────┘
  │
  ▼
┌──────────────┐

└──────────────┘
  │
  ▼
┌──────────────┐     ┌────────┐

```

### 本节要点回顾

|概念|要点|
|---|---|
|**语法制导翻译(SDT)**|给文法产生式附加语义规则,一边分析一边计算|
|**属性**|附加在文法符号上的信息(val、type、code 等)|
|**综合属性**|子→父流动,适合 LR|
|**继承属性**|父→子或兄弟间流动,适合 LL|
|**S-属性定义**|只含综合属性,对 LR 友好|
|**L-属性定义**|综合+受限继承,适合一遍处理|
|**符号表**|记录所有声明过的名字,实现作用域|
|**类型检查**|用 type 属性在树上流动,检查一致性|

#### 一个观念升级

从这节课开始,你应该有一个新的视角:

> **文法不只是语言的"规则书",更是计算的"骨架"**。
> 
> 我们把各种处理(类型检查、代码生成、优化提示、作用域管理……)都挂到这个骨架上,让它们随着分析过程自动完成。

这就是为什么**编译器的设计是"以文法为中心"的**。

### 先问你三个问题

我们已经有了**语法树**(带属性、带类型信息)。现在假设你是编译器的设计者——你要把这棵树变成最终的机器码。

**问题 A**:假设目标机器是 x86,而下次你又要给 ARM、RISC-V 生成代码。你觉得最笨的做法是什么?最聪明的做法是什么?

**问题 C**:如果让你设计一种"中间代码"——既比源语言更接近机器,又不绑死在某个具体机器上——你会让它长什么样?有几个操作数?有没有变量?有没有控制流?

先想一想,尤其是问题 C——这是本节的核心。

### 让我接住你的思考

**问题 A**:

- 最笨:为每种源语言 × 每种目标机器写一个翻译器。n 种语言 × m 种机器 = **n × m** 个翻译器,爆炸。
- 最聪明:定义一种"中间语言",每种源语言翻译到它(n 个前端),每种机器从它翻译出去(m 个后端)。**n + m** 个组件就够了。

这就是为什么 GCC 能支持这么多语言和这么多架构——它有一个叫 **GIMPLE** 的中间表示,LLVM 的叫 **LLVM IR**。**中间代码是编译器"解耦"的支点**。

**问题 B**:多翻译一次,看似低效,其实换来三个巨大好处:

1. **可移植性**:前后端分离,语言和机器互不依赖
2. **优化机会**:在中间层做优化(比如常量折叠、死代码消除),一次优化对所有后端都有效
3. **分析友好**:中间代码结构规整,便于做静态分析

**问题 C**:这个问题我先不揭晓答案,带着你的想法继续看——等会儿你会发现,你如果想"什么能表达源语言所有操作,又不陷入具体机器细节",得到的设计会自然收敛到**三地址码**。

### Part 1:中间表示的几种形式

中间代码(Intermediate Representation, IR)不是唯一的——设计者可以有不同选择。常见的三种:

#### 1. 后缀式(逆波兰式)

你在上节练习题 2 里已经做过了。`a + b * c` → `a b c * +`。

**优点**:结构极简,一个栈就能执行。 **缺点**:没有变量、没有控制流、不适合复杂程序。适合做表达式求值(计算器、JVM 字节码的底层部分),但不适合做完整编译器的主力 IR。

#### 2. 语法树/DAG

直接用带属性的语法树,或者把共享子表达式合并成 DAG(有向无环图)。

比如 `a + a * (b - c) + (b - c) * d`:

- 左 `b - c` 和右 `b - c` 其实是**同一个计算**,DAG 把它合并成一个节点

**优点**:结构化、利于合并共享子表达式。 **缺点**:不够"扁平",离机器还有距离。

#### 3. 三地址码(Three-Address Code, 3AC)⭐

这是主流——我们这节重点讲。

---

### Part 2:三地址码详解

#### 基本形式

每条指令最多**三个地址**(一个目标 + 两个源操作数):

```
x = y op z       （二元运算）
x = op y         （一元运算)
x = y            （赋值)
```

比如 `a + b * c` 翻译成三地址码:

```
t1 = b * c
t2 = a + t1
```

`t1`、`t2` 是编译器自己生成的**临时变量**(temporary),用来存中间结果。

#### 为什么叫"三地址"?

因为每条指令最多涉及三个"地址"(变量名或临时变量)。这个限制看起来任意,其实很有道理:

> 🤔 **想想**:真实机器的二元算术指令(ADD、SUB、MUL)有几个操作数?
> 
> x86 的 `add eax, ebx` 是两操作数(结果覆盖 eax);ARM 的 `ADD R1, R2, R3` 是三操作数。无论哪种,**一条机器指令基本就是一次二元运算**。三地址码就是对齐了这种粒度。

**关键观念**:三地址码是对"机器指令"的抽象——**保留了"一次一运算"的节奏,但去掉了具体寄存器、具体指令集的细节**。

#### 完整的三地址码指令集

除了算术赋值,一个完整的三地址码系统还包括:

```
# 算术和赋值
x = y + z
x = -y
x = y

# 条件和跳转
if x relop y goto L     (relop 是 <, >, ==, !=, <=, >=)
goto L
L:

# 过程调用
param x                 (压参数)
call p, n               (调用 p,n 个参数)
x = call p, n           (有返回值的调用)
return x

# 索引和指针
x = y[i]                (数组取值)
x[i] = y                (数组赋值)
x = &y                  (取地址)
x = *y                  (解引用)
*x = y
```

**你会发现:这基本就是"不带具体指令编码的汇编"**。它有变量名(不是寄存器)、有高级操作(不是指令序列),但控制流已经"扁平化"成 goto 和标签。

#### 三地址码的两种实现

工程上,三地址码不是以字符串形式存在的(那太慢了),而是用数据结构表示:

**(1) 四元组(Quadruple)**:`(op, arg1, arg2, result)`

```
t1 = b * c    →   (*, b, c, t1)
t2 = a + t1   →   (+, a, t1, t2)
```

**(2) 三元组(Triple)**:`(op, arg1, arg2)`,结果用位置引用

```
(0): (*, b, c)
(1): (+, a, (0))   ← 引用第 0 条的结果
```

三元组省了 result 字段,但指令重排就很痛苦(位置变了所有引用都要改)。所以**现代编译器用四元组居多**。

---

### Part 3:用语法制导翻译生成三地址码

这里是语法制导翻译真正展示威力的地方。回忆上节课,我们说**可以把"代码串"作为属性传递**——现在来具体看看。

#### 表达式的翻译

给每个表达式节点 E 两个综合属性:

- **E.place**:这个表达式的值最终存在哪里(一个变量名或临时变量)
- **E.code**:计算这个值所需的三地址码序列

翻译规则:

```
产生式            语义动作
──────────────────────────────────────────────────────
E → E₁ + E₂      E.place = newtemp()
                 E.code = E₁.code || E₂.code || 
                          gen(E.place '=' E₁.place '+' E₂.place)

E → E₁ * E₂      E.place = newtemp()
                 E.code = E₁.code || E₂.code || 
                          gen(E.place '=' E₁.place '*' E₂.place)

E → - E₁         E.place = newtemp()
                 E.code = E₁.code || 
                          gen(E.place '=' '-' E₁.place)

E → ( E₁ )       E.place = E₁.place
                 E.code = E₁.code

E → id           E.place = id.name
                 E.code = ''     (什么都不用算,id 本身就是它的位置)
```

几个关键函数:

- `newtemp()`:生成一个新的临时变量名(t1, t2, t3, ...)
- `gen(...)`:生成一条三地址码指令
- `||`:字符串连接(或指令列表拼接)

#### 走一遍:翻译 `a + b * c`

语法树(左结合解析结果):

```
         E
       / | \
      E  +  E
      |   / | \
      a  E  *  E
         |     |
         b     c
```

(这里 E 能直接推成 id 是简化写法,严格应该是 E → T → F → id,但语义规则一样。)

自底向上算(综合属性):

**节点 `a`**(E → id):

- E.place = "a"
- E.code = ""

**节点 `b`**(E → id):

- E.place = "b"
- E.code = ""

**节点 `c`**(E → id):

- E.place = "c"
- E.code = ""

**节点 `b * c`**(E → E₁ * E₂):

- E.place = t1 ← newtemp()
- E.code = "" || "" || `t1 = b * c`
- = `t1 = b * c`

**根节点 `a + (b * c)`**(E → E₁ + E₂):

- E.place = t2 ← newtemp()
- E.code = "" || `t1 = b * c` || `t2 = a + t1`

最终输出:

```
t1 = b * c
t2 = a + t1
```

结果的"位置"是 t2,表达式的值就在这里。

#### 观察一下

**这整个过程是自动的**。你只要按文法建树、按规则算属性,三地址码就"自动长出来了"。这就是语法制导翻译的美妙。

而且注意:**翻译规则很简单,每条产生式只管自己那一步的翻译**——不需要统揽全局。语法树的结构本身就保证了代码的正确拼接顺序。

---

### Part 4:控制流的翻译——有点挑战

表达式的翻译是"直线型"的。但控制流(if、while)不是。**这里有一个巧妙的技巧,请你先想一想。**

#### 先来一个预热问题

考虑最简单的 if 语句:

```
if (a < b) then x = 1 else x = 2
```

你觉得三地址码应该长什么样?自己在脑子里构造一下,用 goto 和 label。

---

参考答案:

```
        if a < b goto L1
        goto L2
L1:     x = 1
        goto L3
L2:     x = 2
L3:     ...
```

或者更紧凑一点:

```
        if a >= b goto L2   ← 条件取反
        x = 1
        goto L3
L2:     x = 2
L3:     ...
```

#### 问题来了:标签怎么定?

语法分析是自底向上或自顶向下进行的。假设我们在归约 `if-then-else` 这一整条产生式时,**我们手上有三段代码:条件的代码、then 分支的代码、else 分支的代码**。我们要把它们串起来,并在适当位置插入跳转。

问题是:**跳转目标(L1, L2, L3)的值**是什么?我们要跳转到的位置,**是代码在最终线性序列里的地址**,但这个地址此刻还不知道——因为还没生成完!

这就是著名的**回填(backpatching)**问题。

#### 两种解决方法

**方法 1:用符号标签**,先生成,跳转目标写成符号名(L1、L2……),**全部生成完后再统一分配真实地址**。这是最简单的做法。

**方法 2:回填技术(backpatching)**,更巧妙:

- 生成跳转指令时,跳转目标先**留空**(或填 0)
- 把这些"待回填"的指令的地址**记在一个列表里**(truelist、falselist)
- 等到知道目标地址时,**遍历列表,把所有指令的目标字段填上**

回填是 yacc/bison 时代的经典技巧,龙书里讲得很细。但现代编译器(如 LLVM)更多采用"基本块"的结构,直接构造控制流图(CFG),回填问题消失了。

我这里不展开回填的细节(比较繁琐),但你要知道**这个问题的存在**——它是"一遍扫描生成代码"不可避免的挑战。

#### 循环的翻译

`while (B) S` 的三地址码模式:

```
L1:     <B 的代码,结果若假跳到 L2>
        <S 的代码>
        goto L1
L2:     ...
```

`for (i=0; i<n; i++) S` 基本等价于:

```
        i = 0
L1:     if i >= n goto L2
        <S 的代码>
        i = i + 1
        goto L1
L2:     ...
```

**洞察**:所有高级控制结构都能翻译成"条件跳转 + 无条件跳转 + 标签"的底层形式。这就是**语言到机器的第一次"降级"**——从结构化控制流降到扁平的跳转。

---

### Part 5:一个稍完整的例子

源代码:

c

```c
while (i < n) {
    sum = sum + a[i];
    i = i + 1;
}
```

翻译成三地址码:

```
L1:     if i >= n goto L2
        t1 = i * 4           ← a[i] 的字节偏移(假设 int 是 4 字节)
        t2 = a[t1]           ← 数组取值
        t3 = sum + t2
        sum = t3
        t4 = i + 1
        i = t4
        goto L1
L2:     ...
```

几个值得注意的地方:

1. **数组访问 `a[i]` 被拆成两步**:先算偏移 `i * 4`,再做间址取值。**数组这个高级概念被"降级"成了地址算术**。
2. **每个中间结果用临时变量承接**,表达式被**扁平化**。
3. **循环的高级结构变成了两个标签加两条跳转**。

这段三地址码就是编译器前端的最终产物,接下来**优化器和后端**接手工作——我们下节课讲。

---

### Part 6:三地址码之外——现代 IR 一瞥

三地址码是经典教学材料。实际工业编译器用的 IR 更精致:

#### 静态单赋值形式(SSA)

**每个变量只被赋值一次**。上面那段循环改成 SSA:

```
         i₀ = ...
         sum₀ = ...
L1:      i₁ = φ(i₀, i₂)        ← φ 函数合并"从不同路径来的值"
         sum₁ = φ(sum₀, sum₂)
         if i₁ >= n goto L2
         t1 = i₁ * 4
         t2 = a[t1]
         sum₂ = sum₁ + t2
         i₂ = i₁ + 1
         goto L1
L2:      ...
```

那个奇怪的 `φ`(phi)函数表示"这个变量的值取决于从哪条路径到达这里"。

**为什么要 SSA**:变量只赋值一次后,**数据流分析变得极其简单**——看到一个变量的使用,立刻就能追到它唯一的定义点。这让优化变得高效。

**LLVM IR、GCC 的 GIMPLE(在某些阶段)、Rust 的 MIR** 都用 SSA。这是现代编译器的标配。

你不用现在掌握 SSA,但请记得这个名字——下节课讲优化时会再碰它。

---

### 本节要点总结

|概念|要点|
|---|---|
|**为什么要 IR**|前后端解耦、支持优化、便于分析|
|**三地址码**|`x = y op z`,对机器指令的抽象,主流教学 IR|
|**四元组/三元组**|三地址码的具体数据结构|
|**翻译表达式**|用 `.place` 和 `.code` 两个综合属性自动生成|
|**翻译控制流**|降级为 goto + label,存在"回填"挑战|
|**数组/指针**|被降级为地址算术|
|**SSA**|每个变量只赋值一次,现代 IR 主流|

#### 关键观念

**中间代码生成就是"降级"的第一波**:

- 嵌套表达式 → 扁平的三地址序列
- 结构化控制流(if/while/for) → 跳转和标签
- 数组下标 → 地址算术

这一波降级之后,代码已经"几乎像汇编"了——但它还不是汇编,因为还没涉及寄存器分配、指令选择等具体机器细节。这些是**后端**的工作,下节课见。

## 第八节:代码优化

### 开篇:优化的根本原则

先把那个"凭什么敢改"的问题答了,因为它贯穿整节课。

**编译器优化遵循一个铁律**:

> **优化前后的程序必须"可观察行为"等价**。

这里有两个关键词:

**"可观察行为"**——指的是程序对外界的影响:输出、返回值、文件读写、系统调用等。**不包括**"用了哪些指令""用了多少临时变量""执行顺序是否完全一致"这些内部细节。

**"等价"**——**在所有合法输入上,行为相同**。注意:只在"合法"输入上,这是一个微妙但重要的限定。

举个例子解释这个限定:C 语言规定"有符号整数溢出是未定义行为(UB)"。这意味着编译器**可以假设程序员不会让它溢出**。于是下面这种优化就合法:

c

```c
// 原始代码
if (x + 1 < x) { ... }   // 检测是否溢出

// 优化后(编译器"假设 x + 1 不会溢出")
if (false) { ... }       // 永远为假,整个分支被删掉!
```

听起来吓人——真实世界里这确实坑过无数 C 程序员。但规则就是规则:**UB 给了编译器优化的"许可证"**。

所以回答问题 C:**编译器敢改代码,因为它在语言规范划定的"允许范围"内改。超出这个范围的代码,程序员自己负责。**

记住这个原则,接下来所有优化都是在这个铁律下跳舞。

### Part 1:优化的分类

先给你一张地图,不然容易迷失。

#### 按作用范围分

```
局部优化(Local)       → 单个基本块内
     ↓
全局优化(Global)      → 单个函数内(跨基本块)
     ↓
过程间优化(Inter-procedural) → 跨函数
     ↓
链接时优化(LTO)       → 整个程序
```

范围越大,机会越多,但分析成本也越高。**学习难度也是递增的**。

#### 按"动手方式"分

- **机器无关优化**:和目标机器无关,在 IR 上做(常量折叠、死代码消除、公共子表达式消除……)
- **机器相关优化**:和具体机器相关(指令选择、寄存器分配、指令调度、窥孔优化……)

这节课重点讲**机器无关的局部/全局优化**,机器相关的留给下节课(代码生成)。

#### 先介绍"基本块"这个概念

优化的基本单位不是一条指令,而是**基本块(Basic Block)**——

> 一段**顺序执行**的指令序列,**从头进入,从尾离开**,中间没有跳入也没有跳出。

简单说:只要进入第一条,就一定会执行到最后一条。

举例,下面这段三地址码:

```
1:   t1 = a + b
2:   t2 = t1 * 2
3:   if t2 > 10 goto L1
4:   x = 0
5:   goto L2
6: L1: x = 1
7: L2: y = x + 1
```

划分成基本块:

- **B1**:行 1-3(以条件跳转结束)
- **B2**:行 4-5(以无条件跳转结束)
- **B3**:行 6(L1 是跳转目标,必须开新块)
- **B4**:行 7(L2 是跳转目标)

**基本块的划分规则**:

- 第一条指令是**领导者**
- 跳转目标是领导者
- 跳转指令的下一条是领导者
- 每个领导者开始一个新基本块,直到下一个领导者前

#### 控制流图(CFG)

把基本块画成节点,跳转关系画成边,得到**控制流图**:

```
     B1
    /  \
   B2    B3
    \   /
     B4
```

**CFG 是全局优化的骨架**——所有全局分析都在 CFG 上进行。

### Part 2:局部优化(基本块内)

先从最简单的开始。这些优化**只看一个基本块**,不涉及控制流。

#### 优化 1:常量折叠(Constant Folding)

回答你问题 A 的直觉:能在编译期算的,就别留到运行期。

```
t1 = 3 * 4        →     t1 = 12
x = t1 + 5              x = 17
```

甚至:

```
t1 = 3 * 4        →     x = 17    (t1 和 t2 都被消掉)
t2 = t1 + 5             
x = t2
```

**实现**:扫描基本块,遇到两个操作数都是常量的运算,立即求值。

**一个坑**:浮点数的常量折叠要小心——编译期和运行期的舍入规则可能不一致,严格合规的编译器会保留某些浮点运算不折叠,除非开启了 `-ffast-math`。

#### 优化 2:常量传播(Constant Propagation)

```
x = 10            →     x = 10
y = x + 5               y = 15
z = y * 2               z = 30
```

`x` 被赋了常量 10,后面用到 `x` 的地方都替换成 10,然后触发常量折叠的连锁反应。

#### 优化 3:代数化简(Algebraic Simplification)

利用代数恒等式:

```
x + 0   →   x
x * 1   →   x
x * 0   →   0
x - x   →   0
x / 1   →   x
```

有些看似无用,但在其他优化的**连锁反应后**会出现。比如某段代码经过常量传播后变成 `y = x * 1`,这时代数化简就派上用场了。

#### 优化 4:强度削弱(Strength Reduction)

**用便宜的运算代替贵的运算**。典型例子:

```
x = y * 2         →     x = y + y     或     x = y << 1
x = y * 8         →     x = y << 3
x = y / 4         →     x = y >> 2    (无符号或非负整数)
```

硬件上,**位移比乘法快 5-10 倍**(在除不尽的情况下更明显)。现代 CPU 乘法器虽然快了,但这个优化依然重要——尤其是循环体内的乘法。

#### 优化 5:公共子表达式消除(Common Subexpression Elimination, CSE)

回答你问题 B 的核心。

```
x = a + b         →     t = a + b
y = a + b               x = t
z = a + b               y = t
                        z = t
```

**原理**:`a + b` 算一次,后面都复用。

**但——这里有个坑**。你要确保 `a` 和 `b` 在三次计算之间**没变**。如果中间有 `a = something`,那后面的 `a + b` 就不再是同一个 `a + b` 了:

```
x = a + b
a = 5             ← a 变了!
y = a + b         ← 不能复用,这是新的 a + b
```

**这就是为什么 CSE 需要"数据流分析"**:得搞清楚"这个变量从这里到那里,值有没有变"。基本块内分析还算简单,跨基本块就难了——**这正是 SSA 形式如此受欢迎的原因**,后面会再提。

#### 优化 6:死代码消除(Dead Code Elimination, DCE)

**"死"** = 计算结果后面没人用。

```
x = 5             →     (被删除)
x = 10                  x = 10
y = x + 1               y = x + 1
```

第一行的 `x = 5` 立刻被 `x = 10` 覆盖,没人用——死了,删掉。

**更隐蔽的例子**:

```
if (false) {      →     (整个 if 被删除)
    expensive_computation();
}
```

经过常量传播后,条件变成了 `false`,整个分支成了死代码。

#### 优化 7:复制传播(Copy Propagation)

```
x = y              →    x = y          (可能后面也被删)
z = x + 1               z = y + 1
```

`x = y` 之后,接下来用到 `x` 的地方都可以直接用 `y`——然后第一行可能变成死代码被消除。

### Part 3:DAG:局部优化的统一工具

上面七种优化各讲各的,其实**工程上常常用一个统一的数据结构一次性做完**:**DAG(有向无环图)**。

#### 怎么建 DAG

扫描基本块,每条指令建一个节点。**关键**:遇到运算时,**先查 DAG 中有没有"操作符相同、操作数相同"的现成节点**——有就复用,没有才新建。

**例子**:

```
t1 = a + b
t2 = a + b      ← 和 t1 同样的计算!
t3 = t1 * t2
x = t3
```

建 DAG:

```
      *  (t3, x)
     / \
    +   +       ← 其实这两个 + 是同一个节点,因为操作数都是 a, b
    |\  |\
    a b a b      ← a, b 也是共享的叶节点
```

正确的 DAG(合并后):

```
         *  (t3, x)
        / \
        \_/   ← 两条边都指向同一个 + 节点
         |
         +
        / \
       a   b
```

**从 DAG 重新生成代码**时,只需要一条 `+`、一条 `*`,CSE 自动完成。

#### DAG 顺便做的优化

建 DAG 的过程中,许多优化**自然发生**:

- **CSE**:相同的操作复用节点
- **常量折叠**:如果子节点都是常量,直接算出新的常量节点
- **死代码消除**:生成代码时,**没人用的节点不生成**
- **代数化简**:建节点时检查"这个 `+` 的一个操作数是 0 吗?是就直接用另一操作数"

这就是 DAG 的威力:**用一个结构,把几种优化统一在一遍扫描里**。

### Part 4:全局优化——跨基本块

局部优化只在一个基本块内有效。很多真正重要的优化**需要跨基本块**——这就进入了**全局数据流分析**的领域。

这里我不展开讲具体算法,而是**建立框架理解**,这样你看教材时不会迷路。

#### 问题的本质

全局分析要回答的问题形如:

- **到达定值(Reaching Definition)**:"在程序点 p,变量 x 的值可能来自哪些赋值语句?"
- **活跃变量(Live Variable)**:"在程序点 p,变量 x 以后还会被使用吗?"
- **可用表达式(Available Expression)**:"在程序点 p,表达式 `a+b` 是否已经计算过且还没失效?"

这些"信息"在 CFG 上流动,我们要在每个程序点**算出**这些信息。

#### 数据流分析的一般框架

每种分析都可以套入同一个框架:

1. 定义每个基本块的**局部信息**(比如"这个块定义了哪些变量"、"这个块使用了哪些变量")
2. 写出**数据流方程**:一个基本块的"出口信息"是它"入口信息"经过本块处理后的结果
3. **迭代求解**:在 CFG 上反复算,直到所有方程都满足(不动点)

**方向分两种**:

- **前向分析**(信息顺着控制流走):到达定值、可用表达式
- **后向分析**(信息逆着控制流走):活跃变量

我举一个最直观的例子——

#### 活跃变量分析(Liveness Analysis)

**目的**:在每个程序点,知道哪些变量"还活着"(即之后还会被使用)。

**用途**:

- **死代码消除**:给变量赋值,但该变量此后不再被使用 → 这条赋值是死代码
- **寄存器分配**:两个变量在任何时刻都不同时活着 → 可以分配到同一个寄存器

**直觉**:从程序末尾倒着看,每经过一个"使用"就"变活",每经过一个"定值"就"变死"(但要小心:先用后定还是先定后用?)。

看一个迷你例子:

```
B1: x = 1         ← 出口时,x 活;a 不活;b 不活
    a = 2
    b = 3
B2: y = a + b     ← 出口时,x 活;y 活;a 和 b 不再被用
B3: z = x + y     ← 出口时,z 活
    print(z)
```

反着走:

- 在 B3 末尾:print 用了 z,所以 z 活
- 进 B3 前:z 还没被算,但 x 和 y 在 B3 里被用 → x 和 y 活
- B2 末尾:x 活(B3 要用),y 也马上要被算但暂时不算
- 进 B2 前:a 和 b 在 B2 里被用 → a 和 b 活
- B1 末尾:a、b、x 全活
- 进 B1 前:这些变量都还没定义 → 不活

这个信息告诉我们:在 B2 执行完的那一刻,**a 和 b 就死了**——编译器就可以把它们占用的寄存器释放出来分配给别的变量。

### Part 5:循环优化——"性能的主战场"

真实程序 90% 的时间在循环里。**循环优化是优化的重中之重**。

#### 循环不变量外提(Loop Invariant Code Motion)

c

```c
for (i = 0; i < n; i++) {
    x = a * b;        ← 每次循环都算一遍,但 a、b 根本没变!
    arr[i] = x + i;
}
```

优化后:

c

```c
x = a * b;            ← 提到循环外
for (i = 0; i < n; i++) {
    arr[i] = x + i;
}
```

省下了 n-1 次乘法。

**判定条件**:一个表达式在循环内**每次计算结果都相同**(操作数都不随循环变化),就能外提。

#### 归纳变量优化

"归纳变量"指循环中**每次变化量固定**的变量。典型的是 `i++` 里的 `i`。

c

```c
for (i = 0; i < n; i++) {
    a[i] = ...;       ← 内部要算 &a + i*4,涉及乘法
}
```

优化:引入 `p = &a + i*4`,但注意 `p` 每次只增加 4(常数):

c

```c
p = &a;
for (i = 0; i < n; i++) {
    *p = ...;
    p += 4;           ← 变成加法,不再是乘法
}
```

这是**强度削弱在循环中的应用**——把循环内的乘法换成加法。

#### 循环展开(Loop Unrolling)

c

```c
for (i = 0; i < 100; i++) {
    a[i] = b[i];
}
```

每次循环要做:比较 `i < 100`、自增 `i`、跳转。**控制开销**占了相当比例。

展开 4 倍:

c

```c
for (i = 0; i < 100; i += 4) {
    a[i]   = b[i];
    a[i+1] = b[i+1];
    a[i+2] = b[i+2];
    a[i+3] = b[i+3];
}
```

**控制开销降到 1/4**,而且循环体内指令可以更好地被 CPU 流水线并行执行。

**代价**:代码体积变大,指令缓存压力增加。所以展开因子要权衡——编译器通常根据启发式选择。

### Part 6:SSA 让一切优化变简单

还记得上节课提到的 **SSA(静态单赋值)** 吗?这里我再展开一点,你就会明白为什么现代编译器如此推崇它。

#### 回忆 SSA 的核心规则

每个变量只赋值一次。如果源代码里 `x` 被赋值了三次,SSA 里就变成 `x1`、`x2`、`x3`。

#### SSA 如何让优化变简单

以 CSE 为例。**非 SSA 下**,判断 `a + b` 是否能复用,你要问:

> "从上次计算 `a + b` 到这里,`a` 和 `b` 有没有被重新赋值?"

这问题要做**数据流分析**才能答——跨基本块就复杂了。

**SSA 下**,变量只赋一次,所以:

> "`a₁ + b₁` 和这里的 `a₁ + b₁` 操作数完全相同,必然可以复用。"

**一目了然**。数据流分析几乎消失了。

类似地,常量传播、复制传播、死代码消除——在 SSA 下都变得**近乎 trivial**。

所以现代编译器的主流架构是:

```
源代码 → AST → IR(三地址码)→ 转成 SSA → 一堆基于 SSA 的优化 → 转出 SSA → 后端
```

**LLVM 全程 SSA**(包括优化和机器码生成前的阶段),这也是它崛起的重要原因之一。

### Part 7:优化之间的相互作用

一个重要观察:**优化之间会互相"喂料"**。

看这段代码:

c

```c
int x = 10;
int y = x + 5;
int z = y * 0;
print(z);
```

单独看每步:

- 常量传播:`x = 10`,把后面的 `x` 换成 `10` → `y = 10 + 5`
- 常量折叠:`y = 15`
- 常量传播:`z = 15 * 0`
- 代数化简:`z = 0`
- 常量传播:`print(0)`
- 死代码消除:`x`、`y`、`z` 的赋值全没用了 → 全删

最后留下:

c

```c
print(0);
```

**一波连锁反应**。

**所以编译器怎么做?——反复迭代**。每轮跑一遍所有优化,只要还有东西被改变,就再来一轮。直到一轮下来什么都没改了,停止。

这叫**不动点迭代**。GCC 和 LLVM 的 `-O2`、`-O3` 都涉及这种反复打磨。

### Part 8:优化的工程现实

我想你分享几个容易被忽视但很重要的现实。

#### 优化不总是"优化"

**有时候你以为的"优化"反而让代码变慢**:

- 循环展开太多 → 指令缓存溢出 → 变慢
- 过度内联函数 → 代码膨胀 → 变慢
- 改变循环顺序 → 破坏了局部性 → 缓存命中率下降 → 变慢

这就是为什么 GCC 分 `-O0`、`-O1`、`-O2`、`-O3`、`-Os` 等等档位——**高级别不一定最快,要看具体场景**。甚至有时候 `-O3` 比 `-O2` 慢。

#### 最难的优化是"不做不该做的优化"

回到开头那个铁律:**保持可观察行为等价**。

这在实践中比想象的难:

- **副作用分析**:函数 `foo()` 到底改变了哪些全局状态?
- **别名分析**:两个指针可能指向同一个地方吗?如果是,CSE 就不能做。
- **内存序和多线程**:优化能不能跨 `atomic` 操作重排?C++ 的 `volatile` 和内存模型就是约束优化器的。

**一个老笑话**:优化器最大的敌人是指针。因为指针可能指向任何地方,编译器必须保守假设"任何 `*p = ...` 都可能改变任何变量",于是很多优化都做不了。

这也是 Rust 备受推崇的原因之一——**借用检查器提供的别名信息让优化器做出 C 做不了的优化**。

#### 大部分程序的性能,优化器帮不上忙

**最残酷的真相**:

> 编译器优化能把你的代码跑快 2-5 倍,但**算法换对了可以让你的代码快 1000 倍**。

冒泡排序 O(n²) 优化到极致,也快不过 O(n log n) 的快排。**优化器只在你已经选对算法的前提下才有价值**。

所以程序员的优先级是:

1. 选对算法和数据结构
2. 写清晰、简单的代码(编译器优化器最爱)
3. 测量,找到真正的瓶颈
4. 针对瓶颈手工调优

编译器帮你完成第 2 步到第 4 步之间的大量"自动化收尾工作"——这依然很有价值,但不要过度期待。

|**基本块/CFG**|优化的基本单位和骨架|
|**局部优化**|常量折叠/传播、CSE、DCE、代数化简、强度削弱、复制传播|
|**DAG**|局部优化的统一实现工具|
|**数据流分析**|全局优化的理论框架(活跃变量、到达定值等)|
|**循环优化**|不变量外提、归纳变量、循环展开——性能主战场|
|**SSA**|让优化大幅简化,现代编译器主流|
|**工程现实**|优化不总是优化;保守分析很重要;算法 >> 编译器|

#### 一个退一步的反思

前七节课让你看到编译器"如何把程序翻译正确",这节课让你看到"如何让程序跑得快"。但**优化的根本永远在正确性之内**——再聪明的优化,一旦破坏了语义,就是 bug。

这也是为什么**编译器开发者是世界上最谨慎的程序员之一**:一个错误的优化可能影响世界上所有编译过的程序。

### Part 2:指令选择

#### 问题本质

把"一条或多条三地址码"映射到"一条或多条目标机指令"。听起来像"翻译词典",但没那么简单。

#### 一对一的简单情形

```
三地址码            ARM 汇编
─────────────────────────────────
x = y + z           ADD Rx, Ry, Rz
x = y - z           SUB Rx, Ry, Rz
x = y               MOV Rx, Ry
if x < y goto L     CMP Rx, Ry
                    BLT L
```

看起来直接查表。

#### 为什么没那么简单

现实中,**同一件事可以用不同的指令实现,代价不同**。

**例子 1:乘以 2**

```
三地址码:  x = y * 2
```

候选实现:

- `MUL Rx, Ry, #2` (耗时 3-5 周期)
- `ADD Rx, Ry, Ry` (耗时 1 周期)
- `LSL Rx, Ry, #1` (耗时 1 周期,左移 1 位)

**都对,但代价不同**。编译器要选最便宜的。

**例子 2:复合指令**

ARM 有 `MLA`(乘加)指令:`MLA Rd, Rm, Rs, Rn` 意思是 `Rd = Rm * Rs + Rn`。

如果三地址码里是:

```
t1 = a * b
c  = t1 + d
```

天真翻译:

```
MUL R0, Ra, Rb
ADD Rc, R0, Rd
```

聪明翻译:

```
MLA Rc, Ra, Rb, Rd    ← 一条指令顶两条
```

**指令选择器要"看出"这种模式,匹配到 MLA**。这就是所谓的"指令选择"的真正难度——**在指令集的各种复合模式中找最优匹配**。

#### 工程技术:树模式匹配

现代指令选择器的主流做法:把三地址码(或中间码)看成表达式树,把**每条目标机指令也看成一个"模式树"**,在中间表示上做**树匹配**。

就像一个巨大的拼图游戏:

```
中间表示树:            指令模式(MLA):        指令模式(MUL+ADD):
                                               
      +                      +                     +
     / \                    / \                   / \
    *   d                  *   Rn                *   d
   / \                    / \                   / \
  a   b                 Rm   Rs                a   b
                                               
匹配整棵树 → MLA        匹配子树 → MUL          匹配整棵树 → 先 MUL 再 ADD
(1 条指令)              (+ 未匹配)               (2 条指令)
```

编译器用**动态规划**在所有可能的"覆盖方式"中找**总代价最小**的那一种。

**主流工具**:LLVM 的指令选择用的是 **SelectionDAG**,GCC 用的是 RTL。这两者本质都是树/图模式匹配。

#### 窥孔优化(Peephole Optimization)

指令选完之后,再做一道"窥孔"优化——**盯着连续的几条指令(一个"小窗口"),看看能不能进一步优化**。

典型模式:

```
MOV R1, R2          →     ADD R1, R2, #1     (省一条 MOV)
ADD R1, R1, #1

MOV R1, R2          →     (删掉——自己赋给自己)
MOV R1, R2          

ADD R1, R2, #0      →     MOV R1, R2         (代数化简)

JMP L1              →     JMP L2             (跳转到跳转,直接跳目标)
L1: JMP L2
```

窥孔优化**简单粗暴,但非常有效**。而且它可以"捡漏"——很多前面阶段没发现的冗余,在指令级别重新浮现。

### Part 3:寄存器分配 —— 代码生成最硬的骨头

#### 问题陈述

我们有**无限多的变量和临时变量**(t1, t2, t3, ...),但 CPU 只有**有限多的寄存器**(ARM 一般 16 个通用,x86-64 是 16,RISC-V 通用 32)。

**把哪些变量放寄存器?哪些放内存?**

这个问题很重要,因为:

- **寄存器访问**:~1 周期
- **L1 缓存访问**:~3-5 周期
- **内存访问**:~100-300 周期

**多放一个变量到寄存器,循环体可能快几十倍**。

#### 核心洞察:何时两个变量可以"共用"一个寄存器?

回忆第八节的**活跃变量分析**:如果两个变量在整个程序中**从不同时活跃**,它们**可以共享同一个寄存器**。

```
t1 = a + b       ← t1 活
use(t1)          ← t1 最后一次使用,从此 t1 死了
t2 = c + d       ← 这里 t2 开始活,而 t1 已死 → t1 和 t2 可共享寄存器!
use(t2)
```

#### 图着色法——寄存器分配的经典方法

**1950 年代末起,Chaitin 等人**提出了一个漂亮的观察:

> **寄存器分配等价于图着色问题**。

怎么做?

**第一步:构造"冲突图"(interference graph)**

- **每个变量是图中的一个节点**
- **如果两个变量在某一时刻同时活跃,连一条边**

**第二步:用 k 种颜色给图着色**(k = 可用寄存器数)

- 相邻节点颜色不同(冲突的变量不能在同一寄存器)
- 同色节点分配到同一寄存器

#### 看一个具体例子

代码片段:

```
B1:  a = 1
     b = 2           ← a, b 活
     c = a + b       ← a, b, c 活(c 刚定义,a, b 还要再用吗?)
     d = c * 2       ← 假设 a, b 之后不再用,这里 c, d 活
     return d        ← d 活
```

活跃分析(简化):

- 赋 a 后:{a}
- 赋 b 后:{a, b}
- 赋 c 后:{c} (a, b 不再用)
- 赋 d 后:{d}

**冲突图**:

```
a ── b          (a 和 b 同时活过)
                (c 和谁都没同时活过——它定义时 a 和 b 已死)
                (d 和谁都没同时活过)
```

着色:只要 2 种颜色就够了(a 和 b 颜色不同,c 和 d 可以任意)。

**所以最少只要 2 个寄存器**就能实现这段代码。

#### 什么时候颜色不够?

如果冲突图是一个"10 节点的完全图"(每两个节点都相邻),而你只有 4 个寄存器,**无论如何都着不了色**。

这时必须**溢出(spill)**:把某些变量**从寄存器挤出去,放到内存栈帧里**。每次使用时先 `LOAD` 再算,算完 `STORE` 回去。

**代价**:访问速度从 1 周期变成 100 多周期。

#### 选哪个变量溢出?

启发式:

- **被使用次数少**的优先溢出(省的周期少)
- **在循环外**的优先溢出(循环内访问很频繁,溢出代价巨大)
- **已经在栈上有副本**的优先溢出(不用额外 STORE)

工程上用**Chaitin-Briggs 算法**(图着色 + 启发式溢出)或**线性扫描**(LLVM 早期使用,更快但略次优)。现代 LLVM 用的是 **Greedy Register Allocator**,更复杂但生成代码质量更高。

#### 一个你可能想不到的细节

**调用约定(Calling Convention)约束了寄存器分配**。

比如 ARM 和 x86-64 都规定:某些寄存器是"调用者保存"(caller-saved),某些是"被调者保存"(callee-saved)。函数调用前后有不同的语义。

这意味着:**函数调用处会强制某些变量溢出到栈上**——因为调用者保存的寄存器在 call 指令执行后值会变。

这也回答了上节课的思考题 5:**为什么函数调用经常阻断优化**——函数调用点是寄存器分配的"雷区"。

### Part 4:指令调度

#### 问题:CPU 不是串行机器

现代 CPU 的真相:

- **流水线**:一条指令分成多个阶段,多条指令在不同阶段并行执行
- **多发射**:每个周期能启动多条指令(超标量)
- **乱序执行**:CPU 内部可能重排指令,只要结果正确

但**前提是这些指令彼此不依赖**。有依赖的指令必须按序,且有**延迟**。

**例子**:ARM 上,`LDR` 加载指令通常有 **3-4 周期的延迟**。也就是说:

```
LDR R1, [R2]        ← 要 3 周期才能拿到 R1 的值
ADD R3, R1, R4      ← 用 R1,必须等!CPU 流水线停顿
```

但如果中间**塞一些和 R1 无关的工作**:

```
LDR R1, [R2]
ADD R5, R6, R7      ← 不等 R1,马上跑
SUB R8, R9, R10     ← 也不等
ADD R3, R1, R4      ← 此时 R1 已经到位,无停顿
```

**同样三条有用指令,延迟消失了**。这就是**指令调度**:重排指令,填满流水线。

#### 数据依赖

重排不能随意。有三种依赖不能违反:

- **RAW(Read After Write,真依赖)**:A 写 x,B 读 x。B 必须在 A 后。
- **WAR(Write After Read,反依赖)**:A 读 x,B 写 x。B 必须在 A 后。
- **WAW(Write After Write,输出依赖)**:A 写 x,B 写 x。B 必须在 A 后(否则值覆盖错了)。

WAR 和 WAW 其实可以通过"改名"(用不同寄存器)消除——CPU 硬件层面的"寄存器重命名"就做这件事。编译器级别,**SSA 形式天然不存在 WAR 和 WAW**(因为每个变量只赋一次)!

又一次 SSA 的威力:**让调度更容易**。

#### 基本块内的表调度(List Scheduling)

基本思路:

1. 构造**数据依赖图**(DAG):节点是指令,边是依赖
2. 拓扑排序,**每次挑一条"准备好"(所有前驱都完成)的指令**发射
3. 挑选时用**启发式优先级**:关键路径长的优先、延迟大的优先……

这是个 **NP 难问题**的近似解——最优调度是不可行的,启发式够用。

#### 跨基本块的调度

更激进的编译器会做**跨基本块调度**:比如把一条 LOAD 提前到 if 分支之前(只要不违反语义)。

这涉及**推测执行**(speculative execution)、**软件流水**(software pipelining)等高级技术,属于 DSP 和高性能计算编译器的重点——我们不深入,但你要知道名字。

### Part 5:把三件事拼起来——一个小例子

源代码:

c

```c
int sum = 0;
for (int i = 0; i < n; i++) {
    sum = sum + a[i];
}
return sum;
```

三地址码(优化后):

```
        sum = 0
        i = 0
L1:     if i >= n goto L2
        t1 = i * 4
        t2 = a[t1]
        sum = sum + t2
        i = i + 1
        goto L1
L2:     return sum
```

#### 指令选择(ARM 风格,示意)

```
        MOV  R_sum, #0
        MOV  R_i, #0
L1:     CMP  R_i, R_n
        BGE  L2
        LSL  R_t1, R_i, #2      ← i*4 用左移(强度削弱+指令选择)
        LDR  R_t2, [R_a, R_t1]  ← 基址+变址的寻址模式,一条指令
        ADD  R_sum, R_sum, R_t2
        ADD  R_i, R_i, #1
        B    L1
L2:     MOV  R0, R_sum          ← 返回值约定放 R0
        BX   LR
```

注意亮点:

- `LSL` 替代乘法 4
- `LDR [R_a, R_t1]` 一条指令完成 **基址 + 偏移的 LOAD**,不用先算地址

#### 寄存器分配

活跃分析可能得到:

- `sum`、`i`、`n`、`a`(基址)——整个循环都活着,都要分配寄存器
- `t1`、`t2`——短暂活跃,甚至可以共享同一个寄存器

假设我们有足够寄存器(ARM 16 个):

- R_sum = R4
- R_i = R5
- R_n = R6
- R_a = R7
- R_t1, R_t2 可以都用 R8(循环内先用 t1,再算完 t2 后 t1 死,R8 可以立刻复用)

等等,**R_t2 的值要活到 ADD 用完**——得小心!实际上 t1 在 LDR 之后就死了,所以可以:

- R_t1 = R8
- R_t2 = R8(LDR 后覆盖 R8 即可)

#### 指令调度

观察循环体:

```
LSL  R8, R5, #2
LDR  R8, [R7, R8]      ← 等一下,这里 R8 先被写,又在 LDR 里被读!
ADD  R4, R4, R8        ← 用 R8 的 LOAD 结果,有延迟!
ADD  R5, R5, #1
```

`LDR` 之后紧跟 `ADD R4, R4, R8`——**R8 的加载有延迟**,CPU 会停顿。

调度优化:**把 i++ 的 ADD 提前**:

```
LSL  R8, R5, #2
LDR  R8, [R7, R8]      
ADD  R5, R5, #1        ← 移上来,填延迟槽!
ADD  R4, R4, R8        ← 现在 R8 应该到位了
```

3 条有用指令,延迟消失。

#### 看到了什么

在这短短 10 行代码的生成里,**三大任务缺一不可**:

- 指令选择:用 LSL 代替 MUL,用带偏移的 LDR 代替分步
- 寄存器分配:16 个寄存器分给多个变量,临时变量共享
- 指令调度:让 i++ 填满 LDR 的延迟

**每一项做好了**,这个循环在真实硬件上能快好几倍。

 │   符号表建立、类型检查、中间代码生成              │
 └──────────────────────────────────────────────────┘
                        ↓ 优化
 ┌──────────────────────────────────────────────────┐
 │   优化后的三地址码(通常是 SSA)                  │
 │   常量折叠、CSE、DCE、循环优化...                 │
 │   基于 CFG + 数据流分析                           │
 └──────────────────────────────────────────────────┘
                        ↓ 代码生成
 ┌──────────────────────────────────────────────────┐
 │   目标代码(汇编 / 机器码)                       │
 │   指令选择、寄存器分配、指令调度                  │
 └──────────────────────────────────────────────────┘
                        ↓ 链接
 ┌──────────────────────────────────────────────────┐
 │   可执行文件                                      │
 └──────────────────────────────────────────────────┘
```

**整个过程的本质**:把"人类能读懂的文本"**一层一层地"降级"**,同时**在每一层做适合那一层的优化**,最终变成"CPU 能直接执行的电信号"。

---

### 一点超越课程的反思

你可能注意到:编译器这门学科,**从前端到后端,难度的"味道"是不同的**。

- **前端(词法、语法、语义)**:**数学味重**——正则表达式、自动机、文法、形式语言理论。理论清晰,有标准答案,有可证明的正确性。
- **中端(优化)**:**工程和经验味重**——各种启发式、权衡、"通常更快但不保证"。有理论(数据流方程、SSA、不动点),但大量靠试。
- **后端(代码生成)**:**硬件味重**——和具体指令集、微架构、流水线、缓存打交道。每换一个目标机器,后端就要重写大部分。

**前端是"一次性写对的科学",后端是"永远在调优的工程"**。这也是为什么像 LLVM 这样的项目,前端相对稳定,而后端每支持一种新架构都是大工程。

### 编译原理为什么值得学?(回到开篇)

第一节课我说过,学编译原理,是为了理解 EDA 工具链、嵌入式、优化代码……现在你学完,应该能自己回答这个问题:

编译原理给你的不仅是"如何写编译器"的知识,而是三种**通用的思维方式**:

1. **形式化地看语言**:任何"语言"(不只是编程语言,甚至是配置文件、协议、DSL)都可以用文法描述、用自动机识别、用 SDT 处理。你以后做任何需要"解析某种格式"的工作——无论是 JSON、HTML、SQL 还是某个公司内部的脚本——都有了清晰的思路。
2. **翻译即降级**:从高级抽象到低级实现,不是一步到位,而是**设计中间表示**、**逐层翻译**。这个思想在软件架构(微服务间的数据转换)、硬件设计(Verilog → 门级电路)、甚至人机交互(语音 → 意图 → 动作)都适用。
3. **分析与变换分离**:**数据流分析**告诉你"什么可以改",**变换**是"怎么改"。这两件事分开,才能组合出复杂的优化。这也是现代软件工程中 **"先理解,后行动"** 的典范——你做架构重构时,这套思维同样适用。

### 本节要点

|概念|要点|
|---|---|
|**指令选择**|把 IR 映射到目标指令,用树模式匹配找最优覆盖|
|**寄存器分配**|等价于图着色,溢出到栈是代价|
|**指令调度**|重排指令避免流水线停顿,尊重数据依赖|
|**窥孔优化**|指令选择后的"局部捡漏"|
|**调用约定**|约束了寄存器分配,是函数调用优化的边界|
|**SSA 再次登场**|让 WAR/WAW 消失,简化调度|

### 尾声

到此,编译原理这门课我们就讲完了。

九节课,从"什么是语言"到"如何生成机器码",你完整地看到了一门经典的计算机科学课程的内在逻辑。龙书的主要内容——除了一些细节算法(比如 LALR 项目集构造、具体的图着色溢出策略、完整的 SSA 构造算法)——你都接触过了。

我特意没有让你手推一个完整的 LR 分析表,也没让你手写一个图着色寄存器分配器——这些是**实现细节**,你真要深入时,龙书、虎书都会手把手教你。我的目标是让你**理解"为什么是这样"**,而不是"怎么一步步做"。前者是课程给你的终身财富,后者需要时随时可以查。

如果你想继续深入,我的推荐是:

- **想真正写一个编译器**:去做 **Stanford CS143** 或 **CMU 15-411** 的项目作业,或者跟着 Crafting Interpreters 这本书从零写一个语言。
- **想理解现代编译器**:读 LLVM 的源码和文档,或者 Rust 的 MIR 设计。
- **想研究理论**:类型理论(TAPL 这本书)、程序分析(Principles of Program Analysis)。
- **想做硬件方向**:找一本讲 EDA 工具、HDL 综合的书,你会发现**综合器的本质和编译器完全一致**——只是目标从机器码变成了门级电路。
